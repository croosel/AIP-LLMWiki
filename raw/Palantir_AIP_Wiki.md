## 1. AIP 总览

**Palantir AIP (Artificial Intelligence Platform)** 是将 AI 与组织数据和运营连接起来的平台。AIP 旨在驱动跨运营流程的自动化，提供一整套可供组织中所有人使用的工具——从开发者到一线用户。

### 平台定位

AIP 与以下两个核心平台共同构成 Palantir 的操作系统：

| 平台 | 角色 |
|------|------|
| **Foundry** | 数据运营平台（Ontology、数据集成、应用构建） |
| **Apollo** | 自主软件部署的任务控制中心 |
| **AIP** | AI 层，将 LLM 能力嵌入到 Foundry 和 Apollo 中 |

### 交付产品范围

AIP 可驱动全谱系 AI 产品：
- LLM 驱动的 Web 应用
- 使用视觉语言模型的移动应用
- 嵌入本地化 AI 的边缘应用

### 五大关键优势

| 优势 | 说明 |
|------|------|
| **无缝集成** | 与组织在 Foundry 上的现有数据集成，构建可访问多种数据源的 LLM 工作流 |
| **安全与治理** | 访问控制、加密、审计能力，内置治理工具确保 AI 操作的问责性和历史溯源 |
| **模型管理** | 构建、训练和部署 LLM 的综合工具套件，支持多模型选择、版本控制和协作 |
| **可扩展性与性能** | 分布式计算架构，支持大规模数据操作、高性能处理和实时分析，细粒度资源控制 |
| **可解释性与透明度** | 审计追踪、解释和评估工具，助力理解和信任模型决策 |

## 2. AIP 核心功能与应用

### AIP 应用全景表

| 应用 | 描述 | 目标用户 |
|------|------|---------|
| **AIP Assist** | LLM 驱动的支持工具，实时自然语言帮助，上下文感知 | 所有平台用户 |
| **AIP Logic** | 无代码 AI 函数开发环境，点按式使用 LLM + Ontology 数据 | 应用构建者 |
| **AIP Chatbot Studio** | 交互式聊天机器人构建器，支持企业数据和工具 | 应用构建者 |
| **AIP Evals** | LLM 函数测试与评估环境，系统性调试和迭代 | 开发者 |

- **Ontology SDK**: 用 Python/Java/TypeScript 编写 AIP 驱动的应用，内置 AIP Logic 函数访问

### 2.1 AIP Assist

**定义**: LLM 驱动的支持工具，帮助用户导航 Palantir 平台。

**核心特性**:
- 实时自然语言辅助，上下文感知回答
- 键盘快捷键: `Cmd+Shift+U` (macOS) / `Ctrl+Shift+U` (Windows)
- 可从任意平台应用侧边栏打开
- 根据活跃应用不同，回答内容也不同
- 支持多语言

**自定义内容源**:
- 可注册自定义文档类型仓库（通过 Code Repositories）
- 两种增强方式：
  1. 添加到默认 AIP Assist 知识库（与预加载搜索上下文一起使用）
  2. 创建 AIP Assist Chatbots（仅使用指定自定义内容源，聚焦于特定工作流）

---

### 2.2 AIP Logic

**定义**: 无代码开发环境，用于构建、测试和发布由 LLM 驱动的函数。

**核心能力**:
- 使用直观界面工程化 prompts、测试、评估、监控、设置自动化
- 函数可接收输入（Ontology 对象或文本字符串），返回输出或对 Ontology 进行编辑
- 自动化功能：Ontology 编辑可自动应用或暂存供人工审核
- 基于与平台一致的安全模型（用户和函数权限）

**典型用例**:
- 将非结构化输入连接到 Ontology
- 解决排程冲突
- 优化资产性能
- 响应供应链中断

> 详细深入见 [第 3 节](#3-aip-logic-深入)

### 2.3 AIP Chatbot Studio

**定义**: 构建交互式聊天机器人（原称 AIP Agent Studio / AIP Agents）。

**核心特性**:
- 聊天机器人配备企业特定信息和工具
- 由 LLM、Ontology、文档和自定义工具驱动
- 可集成到 Workshop 应用中
- 支持动态、上下文感知的读写工作流
- 可部署为 Functions，在 Automate、Evals、Code Repositories 中使用
- 可通过 Ontology SDK 和平台 API 在外部部署

> 详细深入见 [第 4 节](#4-aip-chatbot-studio-深入)

### 2.4 AIP Evals

**定义**: 专为 LLM 非确定性设计的测试环境，用于评估 AIP Logic 函数、AIP Chatbot 函数或代码编写的函数。

**核心能力**:
- 创建测试用例
- 定义评估函数衡量性能
- 对比不同版本函数的结果
- 在 AI FDE 中通过对话命令创建和运行评估套件
- 构建生产部署前的信心

**核心概念**:
| 概念 | 定义 |
|------|------|
| **Evaluation Suite** | 测试用例、目标函数和评估函数的集合 |
| **Target Function** | 被评估的函数（可同时测试多个） |
| **Evaluation Function** | 比较/评估实际输出与期望输出的方法 |
| **Test Cases** | 定义的输入集和期望输出 |
| **Metrics** | 评估函数的结果，可跨测试用例汇总或逐项比较 |

> 详细深入见 [第 5 节](#5-aip-evals-深入)

### 2.5 AIP Threads

**状态**: Beta

**定义**: 面向企业的一般生产力工具，使用 LLM 完成任务和临时分析。

**核心特性**:
- 无需额外配置或技术专长
- 拖放文档（如 PDF）到界面中
- 从已有文档中选择或选择 AIP Agent
- 支持快速、迭代、跨语言 PDF 分析
- 使用第三方 LLM，符合 Palantir 安全标准

**典型工作流**: 开始 → 上传文档或选择 Agent → 输入请求 → 获取答案/摘要/对比

### 2.6 Palantir MCP

**定义**: Model Context Protocol (MCP) 的实现，使外部 AI IDE 和 AI 代理能够自主设计、构建、编辑和审核 Palantir 平台上的端到端应用。

**两大主要优势**:
1. **代码上下文注入**: MCP 识别当前仓库类型，注入定制化上下文（OSDK 仓库、Python transforms、TypeScript functions），搜索代码片段索引
2. **平台操作能力**: 
   - 搜索 Ontology
   - 安全修改 Ontology
   - 更新 Developer Console 应用
   - 运行 Python transforms 并迭代修复错误（preview_transform 工具）

**Palantir MCP vs Ontology MCP**:
| 维度 | Palantir MCP | Ontology MCP |
|------|-------------|-------------|
| **目标用户** | Ontology 构建者 | Ontology 消费者 |
| **权限** | 可修改 Ontology 类型，不能写数据 | 可安全读写 Ontology 数据 |
| **操作** | 搜索、修改类型、更新应用 | 通过外部 AI 系统执行 Action、写数据 |
| **作用域控制** | - | 通过 Application Scopes 限制 Agent 可执行的操作 |

### 2.7 AIP Analyst

**定义**: 代理式工作流界面，使用自然语言对 Ontology 进行临时分析。

**核心特性**:
- 访问路径: `/workspace/aip-analyst`
- AI 自主搜索 Ontology、创建对象集、转换数据、生成摘要和可视化
- 使用多个搜索词搜索 Ontology 以增加找到相关内容的机会

**可用工具类别**:
| 工具类别 | 功能 |
|---------|------|
| **发现工具** | 发现相关的 Object Type 和 Dataset |
| **分析工具** | 探索和分析检索到的数据 |
| **操作工具** | 执行 Foundry 函数和 Ontology Action |
| **可视化工具** | Vega 图表、地图可视化、Mermaid 图表 |
| **文件处理** | 上传 Excel/CSV/Word/PDF/图像进行分析 |

**附加功能**:
- 导出分析会话为 PDF
- 注意：关闭会话后对话历史不保留，需保存查询和 Vega-Lite 代码

## 3. AIP Logic 深入

### 3.1 核心概念

| 概念 | 说明 |
|------|------|
| **Logic Function** | 接收输入（Ontology 对象、文本字符串等），返回输出（值、对象或 Ontology 编辑） |
| **Blocks** | Logic 函数的组成单元。每个 Block 有特定目的（读写 Ontology、计算、聚合、调用函数、与 LLM 互动） |
| **Debugger** | 运行函数时显示 LLM 思维链（Chain-of-Thought），展示每个步骤和使用的工具 |
| **Evaluations** | 为 Logic 函数编写详细测试 |
| **Ontology 编辑** | 必须通过 Action 发布和调用才能写回 Ontology |
| **自动化** | Logic 函数可自动执行 Ontology 编辑或暂存供人工审核 |

### 3.2 Block 类型

#### 3.2.1 Use LLM Block（核心）

AIP Logic 的核心 Block，由三部分构成：
- **Prompt**: 自然语言指令，建议先写最重要的信息（任务概述），再写数据和工具指导
- **Tools**: LLM 通过工具与 Ontology 交互的机制（LLM 只能请求使用工具，实际调用在用户权限内执行）
- **Output**: LLM 的响应输出

**可用 Tools**:
| Tool | 功能 |
|------|------|
| **Apply actions** | 使用 Action 编辑 Ontology |
| **Call function** | 调用其他 TypeScript/Python/Logic 函数 |
| **Query objects** | 访问指定 Object Type 的属性 |
| **Calculator** | 精确数学计算 |

#### 3.2.2 Apply Action Block

- 确定性地调用 Action，无需经过 LLM Block
- 精确控制参数填写，提高执行速度
- 必须通过 Action 调用 Logic 函数才能写回 Ontology

#### 3.2.3 Execute Function Block

- 调用 Foundry 中的现有函数（TypeScript、Python、Logic 函数）
- 复用已有功能，避免重复实现

#### 3.2.4 Conditionals（条件判断）

- 类似传统编程的 "if-then-else" 语句
- 支持多个分支（when...then）
- 所有分支必须返回一致的输出类型

#### 3.2.5 Loops（循环）

- 对集合中每个元素运行转换和/或 Action
- 输出可以是值列表或 Ontology 编辑
- 可通过 `element` 和 `index` 变量访问当前元素
- 仅操作 List（不操作 Array），会自动插入 Array to List 转换
- 无 Action 的循环迭代将并行执行

#### 3.2.6 Create Variable Block

- 创建可在后续 Block 中使用的变量
- 支持类型: array, boolean, date, double, float, integer, long, object, short, string, timestamp

## 4. AIP Chatbot Studio 深入

### 4.1 核心概念

| 概念 | 说明 |
|------|------|
| **AIP Chatbots** | 配备企业特定信息和工具的交互式助手（原 AIP Agents） |
| **Application State** | 应用变量（原 Parameters），在 Prompt 中定制和控制 LLM 行为 |
| **Instructions & Descriptions** | 编译为 LLM 的原始系统提示，教导 LLM 如何用可用上下文完成任务 |
| **RAG** | 检索增强生成，动态利用外部数据源为 LLM 提供相关信息 |
| **Retrieval Context** | 针对每条消息检索的特定信息，流程：用户消息 → 获取相关数据 → 将数据与消息一起传给 LLM |
| **Tools** | LLM 可用于执行特定操作或检索信息的外部功能/API |
| **Vector Embeddings** | 文本的数值表示，捕获语义含义，用于比较和检索相似文本 |
| **Context Window** | LLM 一次可处理的文本量（含系统 Prompt、对话历史、检索上下文等），超出会报错 |
| **Chatbots as Functions** | 聊天机器人可发布为 Functions，可在 Evaluations、Automate、Code Repositories 中使用 |

**Prompt 编写建议**:
1. 先写最重要的信息（任务概述）
2. 后跟必要数据和工具使用指导
3. LLM 只能访问显式提供的信息

## 5. AIP Evals 深入

**AIP Evals** 是专门处理 LLM 非确定性行为的测试环境。

### 关键概念

| 概念 | 说明 |
|------|------|
| **Evaluation Suite** | 测试用例 + 目标函数 + 评估函数的集合 |
| **Target Function** | 被评估的函数（套件可同时评估多个） |
| **Evaluation Function** | 比较实际输出与期望输出的方法 |
| **Test Cases** | 定义的输入集和期望输出 |
| **Metrics** | 评估函数的结果，按测试用例产生，可汇总或单独比较 |
| **Experiments** | 实验结果可在 Results > Runs 标签页查看 |
| **Metrics Dashboard** | 图表和统计信息的仪表盘视图 |

### 使用场景
- 构建 LLM 函数的信心
- 修改现有实现前验证
- 比较不同模型的表现
- 检查运行间方差

### 5.1 指标仪表板 (Metrics Dashboard)

在 Palantir AIP 架构中，**指标仪表板** 通过 AIP Evals 框架将 AI 的推理逻辑转化为可衡量的定量数据，并以图形化方式呈现，使开发者能够可视化 AI 决策性能。

#### 5.1.1 数据的生成与集成

整个流程从评估套件运行开始，最终以仪表板进行可视化分析：

```
编写测试用例 (Test Cases)
      ↓
创建评估套件 (Evaluation Suite)
  ├─ 指定目标函数 (Target Function)
  └─ 绑定评估函数 (Evaluation Function)
      ↓
运行评估套件 → 模拟真实业务场景
      ↓
结果自动写入平台 Dataset（数据集）
      ↓
Metrics Dashboard 读取 Dataset → 图形化呈现
```

> **关键点**: 评估运行的结果自动写入数据集（Dataset），这是仪表板能够进行可视化分析的基础数据源。

#### 5.1.2 核心性能指标的可视化

指标仪表板通过将复杂的 AI 行为转化为直观图表，支持从多个维度评估决策质量：

**a) 模型横向对比**
- 定量比较不同 LLM（如 GPT-4 与 GPT-3.5）在处理相同 Logic 函数时的表现差异
- 可对比指标：准确率、响应时间、Token 消耗、输出一致性

**b) 决策稳定性分析 (Variance)**
- 展示多次运行之间的偏差/方差
- 可视化 AI 决策的一致性
- 确保在重复相同任务时不会出现不可预测的波动
- 适用于识别"幻觉"或输出漂移的系统性问题

**c) 中间过程质量评估 (Intermediate Parameters)**
- 仪表板不只展示最终结果
- 利用中间参数展示 AIP Logic 中**特定 Block** 的输出性能
- 能够精准定位决策链条中的薄弱环节
- 例如：LLM 提取结构化数据 → 下游函数处理 → 检查每一环的准确率

**d) 本体编辑验证 (Ontology Editing Accuracy)**
- 专门可视化 AI 对本体（Ontology）编辑的准确性
- 确保 AI 驱动的操作符合业务规则
- 检测：错误类型的属性赋值、越权修改、格式违规

#### 5.1.3 使用场景

| 场景 | Dashboard 视图 | 解决的问题 |
|------|--------------|----------|
| 模型选型 | 横向对比图（多模型并行） | 哪个模型最适合当前业务场景？ |
| 版本升级 | Before/After 对比 | 新版本 Logic 函数是否真正提升？ |
| 质量门禁 | 通过/不通过指标阈值 | 是否可以部署到生产环境？ |
| 异常检测 | 方差趋势图 | LLM 行为是否出现漂移？ |
| 审计合规 | 历史指标趋势 | 某次决策基于哪个版本的模型？

## 6. AIP Observability（可观测性）

AIP Observability 是 Workflow Lineage 中的一组能力，提供对 AIP 和 Ontology 工作流执行的可见性。

### 核心组件

| 组件 | 说明 |
|------|------|
| **Metrics** | 性能指标 |
| **Execution History** | 执行历史记录 |
| **Distributed Tracing** | 分布式追踪 |
| **Logging (Service Logs & Debugging)** | 服务日志与调试 |
| **Log Search** | 日志搜索 |

### 集成范围

- AIP 代理、函数、语言模型、自动化、Actions、Ontology
- 是 Palantir 平台级可观测性策略的一部分
- 即使未启用 AIP 也可使用

### 6.1 端到端可观测性监控

指标仪表板不仅覆盖决策质量评估，还集成系统运行性能的实时监控，形成"决策质量 + 运行性能"的双维度可观测性。

#### 6.1.1 资源消耗监测

| 指标 | 监控维度 | 价值 |
|------|---------|------|
| Token 消耗 | 按 Function / Agent / 时间段 | 成本治理、异常预算预警 |
| LLM 调用延迟 | P50 / P95 / P99 | 识别性能瓶颈 |
| 并发吞吐量 | 同时执行的 Logic 函数数 | 容量规划 |
| 错误率 | 按 Block / 按错误类型 | 故障定位 |

> **成本治理链条**: Dashboard Token 消耗 → 对比预算 → 超标告警 → 优化 Prompt 长度或模型选择

#### 6.1.2 实验结果追踪 (Experiments)

在进行 AIP Logic 变更时，通过"实验"机制对比不同版本的运行指标：

```
版本 A (当前生产)         版本 B (候选优化)
      │                         │
      ├─ 共同数据集上运行        ├─ 共同数据集上运行
      │                         │
      └─────→ Metrics Dashboard ←─────┘
                    │
         ┌─────────┴─────────┐
         │                   │
    准确率: 85%          准确率: 92%  ✅
    延迟: 1.2s           延迟: 1.5s
    Token: 450            Token: 380  ✅
```

- Dashboard 以 A/B 对比视图展示
- 量化每一项改进的实际效果
- 支持版本回滚的决策依据

### 6.2 决策逻辑的透明化 — Dashboard + Debugger 协同

指标仪表板与 AIP Logic Debugger 形成互补关系：

| 工具 | 视角 | 输出 | 定位 |
|------|------|------|------|
| **Debugger** | 微观 | LLM 思维链 (Chain-of-thought) | 单个请求的推理过程排查 |
| **Metrics Dashboard** | 宏观 | 多请求的聚合趋势图表 | 系统性模式识别和性能基线 |

**协同工作流**:
1. Dashboard 发现某类测试用例通过率异常下降（宏观趋势）→ 🚨 告警
2. 切换到 Debugger 查看具体失败样例的推理链（微观排查）→ 🔍 定位
3. 修复 Logic 函数 → 重新运行评估套件 → Dashboard 验证指标回升（闭环确认）→ ✅

> 这种"宏观趋势发现 → 微观根因分析 → 修复 → 验证"的闭环，将原本"黑盒"化的 AI 推理过程转变为一个受治理、可审计且可量化的生产流程，确保企业级决策的可靠性。

#### 4. 问责性与文档化 (Accountability & Documentation)

#### 5. 多方利益相关者协作 (Multi-stakeholder Collaboration)

- Checks 和 Checkpoints 实现结构化审批流程
- Checkpoints 使利益相关者能够确认和证明 AI 建议的决策

#### 7. 以人为中心的决策 (Human-centered Decision Making)
- AI 应增强而非替代人类决策
- Ontology 基础的决策支持框架保留人类决策权

- 自动化 + 人类检查点确保关键决策有人类监督
- 退出和回退机制
- 反馈循环捕获人类决策以持续改进 AI 推荐

#### 8. 系统化的负责任 AI (Systematic Responsible AI)
- 平台通过既定工作流引导用户融入负责任 AI 原则
- 负责任 AI 是信任 AI 系统的基础

## 8. 计算使用 (Compute Usage)

### 8.1 Token 机制

- **Token**: LLM 处理文本的基本单位，可以短至一个字符或长至一个完整单词
- Token ≠ 单词（例如 "Palantir" 可能被分解为 "Pal" + "ant" + "ir"）
- 不同模型提供商有不同的 Token 定义
- 计算使用以 **compute-seconds per 10,000 tokens** 计量

**归属规则**:
- 可归属的 Token 使用直接附加到调用应用资源
- 不可归属的使用（如 AIP Assist、Error Explainer）归属于发起使用的用户文件夹

**示例计算**（GPT-4o 模型，24 tokens 输入）:
```
compute-seconds = 24 × 43 / 10,000 = 0.1032
```

## 9. 自带模型 (BYOM)

**Bring Your Own Model**（也称为 "Registered Models"）允许客户连接自己的 LLM 或账户到 AIP 中使用。

### 推荐优先级

1. **首选**: 使用 Palantir 提供的模型（如 OpenAI、Azure OpenAI、AWS Bedrock、xAI、GCP Vertex）或 Palantir 自托管的开源模型（如 Llama）
2. **备选**: 仅在以下情况使用 Registered Models：
   - 法律/合规原因无法使用 Palantir 提供的模型
   - 需要自己微调或定制的 LLM

### 支持范围

| 应用 | 支持 | 接口 |
|------|------|------|
| AIP Logic | 是 | ChatCompletion, ChatCompletionWithVision |
| Pipeline Builder | 是（需 REST endpoint） | ChatCompletion, ChatCompletionWithVision |
| AIP Chatbot Studio/Threads | 是（Streaming 为 Beta） | ChatCompletion, StreamedChatCompletion |
| Workshop AIP Chatbot Widget | 通过 Chatbots 和 AIP Logic 支持 | 不适用 |

### 注册方式

1. 使用 **function interfaces** 注册外部托管的 LLM
2. 实现 `@ChatCompletion()` 和 `@ExternalSystems` 装饰器
3. 在函数逻辑中使用 REST source 的 URL 和 secret 访问外部模型
4. 返回符合 `FunctionsGenericChatCompletionResponse` 格式的响应

### 12.1 排序逻辑

#### 显性权重 — 180 天销量驱动

| 规则 | 细节 | AIP 应用 |
|------|------|---------|
| 销量权重 | 近 180 天线上销量为最强排序因子 | Function: **RankBySales()** → 输入近 180 天销量加权 |
| 信保底价 | 成交价 ≥ 页面价 50% | Conditionals: 若成交价 < 50% 页面价 → 不计入销量 |
| 低价引流 | 新店&新品初期可用，但不能成交不卖 | Logic: 自动检测最低价是否可实际成交 |
| 买家数 | 90 天内同一买家多笔订单计为 1 | Ontology 聚合: deduplicate by buyer_id within 90 days |
| 180 天最低价 | 页面价、折扣价、成交价取最低 | Conditionals: 取三者 min |

**定价策略矩阵**:
| 阶段 | 策略 | 操作 |
|------|------|------|
| 冷启动 | 低价引流 | 接近成本底价，吸引首单交易积累销量 |
| 成长期 | 活动营销 | 新贸节/采购节 10-20% OFF，提前发布高价品预热 |
| 成熟期 | 高价引流 | 发布高价 → 全站推获取基础流量 → 活动折扣获取免费流量 |
| 稳定期 | 价值定价 | 突出认证 / 无忧退 / 3天发货等服务力差异化 |

#### 隐性权重 — 相关性匹配（AI Mode 全局匹配）

| 维度 | 规则 | AIP 映射 |
|------|------|---------|
| 标题 | 128 字符，匹配买家搜索第一要素 | Ontology: Product.title（索引字段） |
| 动态标题 | 根据买家偏好实时展示多套 AI 标题 | Logic: DynamicTitle() — LLM 生成多套标题 |
| 商品卖点 | 2000 字符，进入搜索索引（最大 SEO 金矿） | Ontology: Product.selling_points（全文索引） |
| 商品属性 | SKU/属性全部进入搜索匹配 | Ontology: 所有 Property 字段参与搜索 |
| 千人千面 | 不同买家看到的标题不同 | Logic: 输入买家画像 → 输出匹配标题 |

## 15. AIP-Derived Agent 并行架构

> **来源**: 基于 AIP "Sense → Reason → Act" 的依赖拓扑分析
> **验证日期**: 2026-05-19
> **参考**: [New_Product_Workflow_Analysis.md](New_Product_Workflow_Analysis.md) · [data/aggregate_report_2026-05-19.md](data/aggregate_report_2026-05-19.md)

### 15.1 第一性原理：拆 vs 不拆

> **核心问题**: 多步骤任务应该由一个 Agent 串行执行，还是拆成多个 Agent 并行？

**判定法则 — 依赖拓扑分析法**:

```
                     ┌─────────────┐
                     │  数据入口    │ ← 唯一的前置依赖
                     │  (Phase 1)  │
                     └──┬──┬──┬──┬─┘
                ┌───────┘  │  │  │  └───────┐
                ▼          ▼  ▼  ▼          ▼
          ┌─────────┐ ┌─────────┐ ┌─────────┐
          │ 子任务A  │ │ 子任务B  │ │ 子任务C  │  ← 三者零依赖
          └────┬────┘ └────┬────┘ └────┬────┘
               └───────────┼───────────┘
                           ▼
                     ┌──────────┐
                     │  聚合点   │ ← 最终的串行节点
                     │ (Phase 3)│
                     └──────────┘
```

| 判断条件 | 串行 | 并行 |
|---------|------|------|
| 子任务之间有数据依赖 | ✅ | ❌ |
| 子任务之间零依赖且各自独立 | ❌ | ✅ |
| 子任务需要一致性闭环（如标题↔卖点↔主图） | ✅ 绑在同一Agent | ❌ 拆开会增加拼接成本 |

### 15.2 两种拆分方案的实证对比

**实验**: 5 个产品 × 3 项产出（标题 + 卖点 + 主图）= 15 个任务单元

| 架构 | 并行度 | 墙钟时间 | 协调成本 | 一致性 |
|------|--------|---------|---------|--------|
| **A. 单Agent串行** | 0 | ~300s | 零 | ✅ 强 |
| **B. 按任务类型拆3 Agent** | 3路 | ~120s | 高（需主Agent缝合标题↔卖点↔主图） | ⚠️ 弱 |
| **C. 按产品拆5 Agent** ⭐ | 5路 | **~60s** | 低（每Agent自闭环） | ✅ 强 |

**结论**: 按「产品」拆分的 Agent 架构同时达成最高并行度、最低协调成本和最强一致性。

### 15.3 3-Phase 流水线模板

```
Phase 1 (主Agent串行)      Phase 2 (N Agent并行)          Phase 3 (主Agent聚合)
┌──────────────┐        ┌─────────────────────┐        ┌──────────────┐
│ 数据筛选+结构化 │   ──→  │ Agent 1..N 同时启动    │   ──→  │ 一致性检查   │
│ 输出JSON文件   │        │                     │        │ 汇总报告     │
└──────────────┘        │ 每个Agent入参独立      │        └──────────────┘
                        │ 每个Agent输出独立文件  │
                        └─────────────────────┘
```

**适用范围**: N 个独立产品/实体 × M 项产出的流水线作业。N 可线性扩展。

## 16. 产品改造工作流（下架→修改→重上）

> **目标**: 将低权重的「普通品」改造为「定向甄品」，以最小时间成本获取最大自然流量。
> **验证日期**: 2026-05-19

### 16.1 核心洞察

三个平台机制在同一次操作中被触发：

```
下架普通品 ──→  按定向征品要求修改  ──→  重新发布
     │                │                    │
     ▼                ▼                    ▼
 清掉无权重品     匹配买家需求          触发新品90天扶持
                 (定向甄品)             (自然流量放大)
```

### 16.2 四种上品方式效率对比

| 方案 | 操作路径 | 单品耗时 | 日耗时(5个) | 新品标记 | ID变化 |
|------|---------|---------|------------|---------|--------|
| A · 全新直上 | 从零创建 | 18-22 min | 90-110 min | ✅ | 新ID |
| B · 旧品编辑重上 ⭐ | 下架→修改→重新发布 | 8-12 min | **40-60 min** | ✅ | 保留原ID |
| C · 裂变克隆 | 复制已有→改关键字段 | 6-10 min | 30-50 min | ✅ | 新ID |
| D · 在线编辑 | 不下架直接修改 | 5-8 min | 25-40 min | ❌ | 原ID |

> 方案 B 是黄金方案：同时获得新品流量 + 征品匹配 + 保留产品ID + 2-3倍效率提升。

### 16.3 每日 SOP（55 分钟 / 5 个产品）

| 步 | 耗时 | 操作 |
|----|------|------|
| 1 | 5 min | MA后台 → 产品参谋 → 导出普通品清单 → AI 交叉匹配征品要求 |
| 2 | 5 min | 筛出5个候选（优先：高曝光零询盘 > 有TM信号 > 匹配征品类目） |
| 3 | 40 min | 逐一：下架 → 改标题(128字符含品牌+征品核心词+属性+场景) → 换主图(场景化) → 填卖点(2000字符) → 匹配征品属性 → 重新发布 |
| 4 | 3 min | 确认5个新品发布成功，检查无违规 |
| 5 | 2 min | 标记"待观察入橱窗"，7天后数据参谋回测 |

### 16.4 自动化辅助（可提速至 25-30 min/天）

| 环节 | AI 辅助 | 节省 |
|------|---------|------|
| 普通品筛选 | AI 读数据参谋导出表 → 按征品匹配度排序 | -15 min |
| 标题生成 | AI 根据原品+征品要求 → 3方案 | -2 min/个 |
| 卖点生成 | AI 读行业热词 → 2000字 | -5 min/个 |
| 主图生成 | AI 白底图 → 场景图 | -3 min/个 |
| 发布确认 | AI 检查5新品 → 差异列表 | -2 min |

### 16.5 月效果预测（基于 126 普通品的店铺）

| 周 | 累计改造 | 自然曝光 | 变化 |
|----|---------|---------|------|
| W1 | 25 | 3,500/周 | +21% |
| W2 | 50 | 3,800/周 | — |
| W3 | 75 | 4,200/周 | — |
| W4 | 100 | **4,600/周** | **+59%** |

> ⚠️ 产品等级评估有 15-30 天滞后。改造品 30 天后才能被评定为「优品」。W1 应同步启动 P4P $20/天给改造品测款。

### 16.6 风险矩阵

| 风险 | 缓解 |
|------|------|
| 重复铺货被判定 | 改标题 ≥50% + 换主图 + 差异化属性 |
| 征品类目变更 | 每周复查 MA 后台征品更新 |
| 编辑失误致审核不通过 | AI 辅助填充 + 批量检查 |
| 操作过快被限流 | ≤5个/天 + 间隔 ≥1h/批次 |
