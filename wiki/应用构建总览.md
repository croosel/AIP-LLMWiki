# 应用构建总览

## 定义

Palantir 应用构建体系让**多元化的构建者社区**（从零代码用户到专业开发者）都能基于 Ontology 创建操作应用。所有工具共享 Foundry 的安全、血缘、数据和计算原语，让团队专注于交付操作能力而非管理基础设施。

> 关键设计原则：每个工具都**持续、安全地丰富同一套数据和模型资产（Ontology）**——知识随工作流扩展而累积。

## 核心组成

### 应用构建 vs 工作流构建 vs 开发者工具

```
Foundry 应用构建体系
├── 应用构建工具
│   ├── Workshop — 面向对象、零/低代码、Widget 驱动
│   └── Slate — 拖拽式、CSS/JS 自定义、外部数据源
├── 工作流构建与管理
│   ├── Workflow Lineage — 工作流可视化与调试
│   ├── Automate — 条件→效果自动化引擎
│   ├── Solution Designer — 架构图 + AIP Architect
│   └── Use Cases → Portfolios（进化中）— 项目管理
└── 开发者工具（参见 [[开发者工具链总览]]）
    └── OSDK / Functions / Code Workspaces / MCP / Custom Endpoints
```

## 核心属性

### 1. Workshop — 面向对象应用构建

**定位**：利用 Ontology 的语义原语（Objects/Links）和动力原语（Actions/Functions），快速交付高度交互的桌面和移动应用。

**能力层次**：
| 层级 | 能力 | 典型用户 |
|------|------|---------|
| 零代码 | Widget 拖拽 + Ontology 绑定 | 业务运营人员 |
| 低代码 | Variables + Events + 条件逻辑 | 应用构建者 |
| 代码扩展 | Function-backed Widget / OSDK | 开发者 |

**Workshop 超越 Dashboard**：不只是被动可视化——用户可在应用中执行 Actions、编辑数据、触发业务操作，形成真正的操作闭环。

### 2. Slate — 灵活定制应用

**定位**：让开发者用 HTML/CSS/JavaScript 完全控制 UI，同时无缝集成 Ontology。

**使用场景**：需要高度定制品牌体验或公开访问（无需 Foundry 账户）时。更多对比见 [[Ontology应用生态#Workshop vs Slate 选择指南]]。

### 3. Workflow Lineage — 工作流可视化

**定位**：交互式工作空间，理解和管理应用及其底层流程。

**核心能力**：
- **溯源图**：可视化 Object/Action/Function/LLM/Application 之间的关系
- **属性溯源**：深究单个属性的来源和消费者
- **Widget/Variable 溯源**：Workshop 内部的变量流动可视化
- **升级工具**：批量重构和升级工作流资源
- **Global Branching 支持**：在分支中检验变更再合并

**原名**：Workflow Builder → 更名后定位更清晰：不只是"构建"，更是"理解、管理、调试"。

### 4. Automate — 业务自动化引擎

**定位**：取代 Object Monitoring，成为平台所有业务自动化的单一入口。

**三种条件**：
| 条件类型 | 说明 | 示例 |
|---------|------|------|
| Time-based | 按时间表触发 | "每周一 9AM" |
| Object Data | 基于 Ontology 对象数据变化 | "新增 high priority Alert" |
| 组合条件 | Time + Object | "每周检查一次 + 优先队列中是否有新对象" |

**四种效果**：
- **Action effects** — 执行 Ontology 对象创建/修改/删除
- **Logic effects** — 执行 AIP Logic 函数
- **Function effects** — 执行 TypeScript/Python 函数
- **Notification effects** — 发送通知给用户/组

**Fallback effects**：主效果失败时的备用方案（记录/通知/替代工作流）。

### 5. Solution Designer — 架构设计与 AIP Architect

**定位**：创建 Palantir 解决方案的**架构表示**。

**核心能力**：
- 绘制架构图（对象/应用/数据流/依赖关系）
- 从空白画布或现有数据血缘图开始
- **AIP Architect 插件**：LLM 驱动的实现助手

**AIP Architect 流程**：
```
输入需求 → 生成 Workflow Plan（Implementation Plan）
              ↓
         Walkthrough 引导 → 逐步实现
              ↓
    在 Workshop/Automate 等工具中构建实际工作流
```

### 6. Use Cases → Portfolios（演进中）

Use Cases 已进入 Sunset 阶段，推荐使用 Portfolios 替代。它提供：
- 操作资源的统一管理界面
- 文件系统 + Ontology 管理双视图
- Active / Experimental / Deprecated 状态标签
- 应用/对象类型/操作类型/数据源一页聚合

## 业务用途

### 知识累积效应

所有工具共享 Ontology → 每个新应用/新工作流**都在丰富同一套数据资产** → 知识不流失，能力累积增长。

### 构建者分层策略

| 角色 | 首选工具 | 备用 |
|------|---------|------|
| 业务运营人员 | Workshop（零代码 Widget） | Object Explorer |
| 应用构建者 | Workshop（Variables+Events） | Slate |
| 前端开发者 | Slate（CSS/JS 自定义） | OSDK React |
| 数据科学家 | Code Workspaces（Jupyter） | Functions |
| 工作流设计师 | Solution Designer + Automate | Workflow Lineage |

### Workshop 到 LILIS 的映射

Workshop 的核心架构可以直接映射到 LILIS 运营仪表盘：

```
Workshop                    LILIS Dashboard
Module                      → 页面（P4P监控页 / 产品管理页 / 询盘页）
Layout                      → 布局（Tab切换 / 侧边栏）
Widget                      → 组件（表格 / 图表 / 过滤器）
Variable                    → 状态管理（当前选中产品 / 日期范围）
Event                       → 交互（点击行→打开详情 / 按钮→执行操作）
Action                      → API 调用（调价 / 上下架 / 修改信息）
Module Interface            → 页面参数（URL参数：?product=LIL-0001）
```

### 运营工作流示例

以 P4P 自动调价为例：

```
Automate Condition: 时间条件 "Every 6 hours"
  ↓
检查 P4P Campaign 预算‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾
  │
  ├── 预算 < 10% → Action: Pause Campaign + Notification: Slack
  ├── 消耗率 > 目标 120% → Action: Reduce Bid 20%
  └── 消耗率 < 目标 80% → Action: Increase Bid 10%
```

## 常见问题

**Q: Workshop 和 Slate 的核心区别？**
A: Workshop 面向对象 + 配置化（无需代码），Slate 拖拽 + CSS/JS 自定义（需要前端技能）。Workshop 更快，Slate 更灵活。

**Q: Automate 和 Functions 的区别？**
A: Automate 管理"何时做"（条件→触发），Functions 定义"做什么"（业务逻辑）。两者协同：Automate 条件满足 → 触发 Action/Logic/Function 效果。

**Q: Solution Designer 只是画图工具吗？**
A: 不——它是实现工具。AIP Architect 能将架构图直接转化为 Implementation Plan 并引导实现。它连接了"设计"和"构建"。

**Q: Use Cases 已 Sunset，用什么替代？**
A: Portfolios。Use Cases 仍然可用但不再开发新功能。
