# 本体增强生成 (OAG)

## 定义
**本体增强生成 (Ontology-Augmented Generation, OAG)** 是 Palantir AIP 独有的 AI 推理模式。不同于传统 RAG（检索增强生成）向 LLM 注入文本片段，OAG 直接将本体中的**结构化对象（Objects）** 注入 LLM 上下文，实现基于真实业务实体的确定性推理。

## 核心属性
| 属性 | 值 |
|------|-----|
| 英文全称 | Ontology-Augmented Generation |
| 对比对象 | RAG (Retrieval-Augmented Generation) |
| 注入内容 | 结构化对象（非文本片段） |
| 核心优势 | 确定性推理，消除幻觉 |
| 语义基础 | 本体的名词（对象类型）+ 动词（行动） |

## 业务用途

### OAG vs RAG 对比
| 维度 | RAG | OAG |
|------|-----|-----|
| 数据源 | 文档/文本库 | Ontology 结构化对象 |
| 注入方式 | 向量检索文本片段 | 直接绑定对象+属性+关系 |
| 语义理解 | 依赖嵌入相似度 | 基于本体预定义语义 |
| 幻觉风险 | 高（文本片段可能不相关） | 低（对象是事实源） |
| 推理边界 | 开放（互联网范围） | 封闭（CWA 约束） |

### OAG 的三层语义
1. **名词识别**：AI 理解「订单」「患者」「产品」是本体中的 Object Type
2. **动词执行**：AI 知道「取消订单」「出院」对应本体中定义的 Action
3. **关系感知**：AI 理解「转诊单」与「患者」之间的 Link 关系

### 确定性推理机制
- LLM 基于具有**预定义属性**和**类型化关系**的对象进行推理
- 每个对象有明确的 Schema 约束，AI 不能凭空创造新属性
- 本体编辑需经过 Action 内置的业务规则验证

## 外贸应用
> 参见 [[Ontology]] · [[AIP神经符号架构]] · [[LILIS数据模型v2]]

对 LILIS 的映射：
- LILIS 7 表 60+ 字段的数据模型即 OAG 的「本体」
- AI 分析 P4P 投放效果时，不是检索文档，而是直接查询「Product」对象关联的「P4PCampaign」和「Keyword」对象
- 标题优化建议基于确定的产品属性（Shape、Material、Occasion），而非泛化文本

## 常见问题
**Q: OAG 是否完全消除了幻觉？**
A: OAG 通过结构化绑定大幅降低幻觉，但不能说 100% 消除。LLM 仍有概率在自由文本生成中出现偏差。AIP 通过 Evals 评估套件和 Action 验证规则做最后防线。

**Q: OAG 与传统数据库查询有什么区别？**
A: 传统查询返回原始数据，需要人工解读。OAG 让 LLM 理解数据的语义并自主推理，例如从「库存 < 安全阈值」推理出「需要下补货订单」并触发相应 Action。
