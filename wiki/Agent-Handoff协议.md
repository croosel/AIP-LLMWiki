# Agent-Handoff 协议

> 本协议定义了当工作在 Agent 之间交接时必须遵循的标准流程和信息格式。  
> 所有 AI Agent（LLM Wiki Agent、QoderWork Agent、Cursor、Claude 等）在无法独立完成任务或需要其他 Agent 接力时，**必须**按本协议生成 Handoff 信息。  
> 本协议与 [[Agent工作流协议]] 配套使用，共同构成知识库的多 Agent 协作基础设施。

---

## 一、Handoff 触发条件

### 1.1 必须触发 Handoff 的场景

| 场景 | 说明 | 典型交接路径 |
|------|------|-------------|
| **任务范围越界** | 当前任务超出了 Agent 角色定义的权限范围 | 任何 Agent -> Human Editor |
| **领域不匹配** | 需要不同领域的专业知识（如技术问题需要业务视角） | LLM Wiki <-> QoderWork |
| **审批需求** | 需要 Human Editor 审批内容或决策 | 任何 Agent -> Human Editor |
| **冲突无法解决** | 命名冲突、内容冲突等无法按规则自行解决 | 任何 Agent -> Human Editor |
| **质量检查发现问题** | 断链、孤儿页面等需要专门修复 | QoderWork -> LLM Wiki |
| **原始资料处理完毕** | raw/ 新资料已转化完成，需要通知后续处理 | LLM Wiki -> QoderWork |
| **批量操作完成** | 大规模页面修改/创建后需要健康检查 | LLM Wiki -> QoderWork |

### 1.2 可选触发 Handoff 的场景

| 场景 | 说明 |
|------|------|
| **发现新术语** | 新术语已添加到术语索引，但需要其他 Agent 在相关页面中补充引用 |
| **内容待深化** | 当前页面框架已建好，但某个章节需要更深入的专业内容 |
| **跨 Agent 依赖** | 一个 Agent 的工作依赖另一个 Agent 的中间产出 |
| **建议性优化** | 发现可改进之处但不属于自身职责 |

## 二、Handoff 信息格式

### 2.1 标准 Handoff 消息结构

每次 Handoff 必须包含以下结构化信息：

```markdown
## Handoff 消息

### 基本信息
- **发起 Agent**: [Agent 名称 + 版本]
- **接收 Agent**: [目标 Agent 角色]
- **时间戳**: YYYY-MM-DD HH:MM (时区)
- **紧急程度**: P0（阻断）/ P1（重要）/ P2（普通）/ P3（建议）

### 已完成工作
- [描述已完成的具体工作内容]
- 新建页面：[页面列表，含文件路径]
- 编辑页面：[页面列表，含修改摘要]
- 新增链接：[数量及主要链接]
- 新增术语：[术语列表]

### 待处理事项
- [ ] [具体待办事项 1]
- [ ] [具体待办事项 2]
- [ ] [具体待办事项 3]

### 关键决策记录
| 决策 | 选项 | 选择 | 理由 |
|------|------|------|------|
| [决策描述] | [可选方案] | [最终选择] | [选择原因] |

### 上下文说明
[任何接收 Agent 需要了解的背景信息，包括：]
- 为什么需要交接
- 遇到了什么障碍或限制
- 当前知识库的整体状态

### 相关文件清单
| 文件 | 操作 | 说明 |
|------|------|------|
| wiki/xxx.md | 新建 | [说明] |
| wiki/yyy.md | 编辑 | [修改内容摘要] |
| raw/zzz.md | 只读引用 | [来源说明] |

### 预期结果
[描述期望接收 Agent 完成什么工作，达到什么状态]
```

### 2.2 紧急程度定义

| 级别 | 定义 | 响应时间 | 示例 |
|------|------|----------|------|
| **P0 阻断** | 知识库结构损坏或出现严重错误 | 立即 | 大量断链、页面内容丢失、schema 冲突 |
| **P1 重要** | 影响知识库质量但不阻断使用 | 当日内 | 多个 stub 页面待补充、关键页面缺反向链接 |
| **P2 普通** | 常规工作交接 | 下次执行时 | 批量新页面创建后的健康检查 |
| **P3 建议** | 优化建议，非必须执行 | 酌情处理 | 标签优化建议、页面结构改进建议 |

### 2.3 简化 Handoff（适用于低复杂度交接）

当交接内容简单（涉及 <= 3 个页面、无决策记录）时，可使用简化格式：

```markdown
## 简要 Handoff

**发起**: [Agent 名称] | **接收**: [目标 Agent] | **级别**: P2/P3

**已完成**: [一句话概述]
**待处理**: [一句话概述]
**涉及页面**: [[页面1]]、[[页面2]]
**备注**: [补充说明]
```

## 三、状态快照协议

### 3.1 状态快照的作用

当工作从一个 Agent 转移到另一个 Agent 时，发起 Agent 应生成一份**状态快照**，记录当前知识库的相关状态，使接收 Agent 能快速了解上下文。

### 3.2 快照内容

```markdown
## 知识库状态快照 — YYYY-MM-DD

### 整体概况
- **Wiki 页面总数**: N
- **Stub/待完善页面**: N（占 X%）
- **Raw 目录文件数**: N（已处理 M 个）
- **当日新建页面**: [列表]
- **当日编辑页面**: [列表]

### 当前工作状态
- **正在处理的任务**: [描述]
- **任务进度**: [百分比或阶段描述]
- **阻塞因素**: [如有]

### 已知问题
| 问题 | 影响范围 | 优先级 | 状态 |
|------|----------|--------|------|
| [问题描述] | [影响页面/区域] | P0-P3 | 待处理/进行中 |

### 待处理队列
1. [任务描述] — 预计工作量：[估计]
2. [任务描述] — 预计工作量：[估计]

### 最近变更日志
| 时间 | Agent | 操作 | 涉及页面 |
|------|-------|------|----------|
| HH:MM | [Agent] | [新建/编辑/删除] | [[页面]] |
```

### 3.3 快照生成时机

| 时机 | 触发方 | 快照范围 |
|------|--------|----------|
| Handoff 交接时 | 发起 Agent | 与交接相关的局部状态 |
| 每日工作结束时 | 最后操作的 Agent | 当日全量变更 |
| 批量操作完成后 | 执行批量操作的 Agent | 批量操作涉及的范围 |
| Human Editor 请求时 | 当前活跃 Agent | 全量状态 |

## 四、一致性检查点

### 4.1 接收 Agent 的检查义务

接收 Handoff 的 Agent **必须**在处理交接内容前执行以下验证：

#### 检查点 1：Handoff 信息完整性
- [ ] Handoff 消息包含所有必要字段（基本信息、已完成工作、待处理事项）
- [ ] 涉及的文件清单与实际修改的文件一致
- [ ] 紧急程度标注合理

#### 检查点 2：已完成工作的验证
- [ ] Handoff 中列出的新建页面确实存在
- [ ] Handoff 中列出的编辑页面的修改内容与描述一致
- [ ] 新建页面符合 [[Agent工作流协议]] 的页面创建规范（frontmatter、结构、命名）

#### 检查点 3：链接一致性验证
- [ ] 新建页面的 `[[wikilinks]]` 全部解析正确
- [ ] 反向链接已正确更新
- [ ] 无新增断链

#### 检查点 4：标签与术语一致性
- [ ] 新页面使用的标签符合标准分类体系
- [ ] 新增术语已同步更新到 [[术语索引]]
- [ ] 无术语定义冲突

### 4.2 检查不通过的处理

当一致性检查发现问题时：

| 问题类型 | 处理方式 |
|----------|----------|
| Handoff 信息不完整 | 记录缺失项，基于已有信息继续工作，在下次 Handoff 时补全 |
| 新建页面不合规 | 修复格式问题后继续处理，记录在质量检查报告中 |
| 断链 | 创建 stub 页面或修复链接，记录在质量检查报告中 |
| 严重结构问题 | 生成 P0 Handoff 消息，通知 Human Editor |

### 4.3 检查确认格式

接收 Agent 完成检查后，输出确认信息：

```markdown
## Handoff 接收确认

**接收 Agent**: [名称]
**原始 Handoff 时间**: YYYY-MM-DD HH:MM
**接收时间**: YYYY-MM-DD HH:MM

### 检查结果
- Handoff 信息完整性: PASS / FAIL（说明）
- 已完成工作验证: PASS / FAIL（说明）
- 链接一致性: PASS / FAIL（说明）
- 标签术语一致性: PASS / FAIL（说明）

### 发现问题
[如有问题，列出详情]

### 接管计划
[描述接收 Agent 接下来的工作计划]
```

## 五、典型 Handoff 场景示例

### 5.1 LLM Wiki Agent -> QoderWork Agent（批量创建后健康检查）

```markdown
## Handoff 消息

### 基本信息
- **发起 Agent**: LLM Wiki Agent v0.2.0
- **接收 Agent**: QoderWork Agent
- **时间戳**: 2026-06-08 14:30 (Asia/Shanghai)
- **紧急程度**: P2

### 已完成工作
- 从 raw/Palantir_Docs_Research_20260520.md 提取并创建了 5 个新 wiki 页面
- 新建页面：[[Workshop详解]]、[[Object-Views构建]]、[[Object-Explorer]]、[[链接类型深度]]、[[Developer Console详解]]
- 在 [[主页]] 导航表中添加了 5 个条目
- 在 [[术语索引]] 中注册了 12 个新术语

### 待处理事项
- [ ] 全量断链扫描（5 个新页面可能与现有页面有未发现的关联）
- [ ] 验证 [[术语索引]] 新增条目与 schema 术语表的一致性
- [ ] 检查 5 个新页面的反向链接是否完整

### 相关文件清单
| 文件 | 操作 | 说明 |
|------|------|------|
| wiki/Workshop详解.md | 新建 | Workshop 构建器详解 |
| wiki/Object-Views构建.md | 新建 | 对象视图模板 |
| wiki/Object-Explorer.md | 新建 | 对象浏览工具 |
| wiki/链接类型深度.md | 新建 | 链接基数与衍生属性 |
| wiki/Developer Console详解.md | 新建 | 开发者控制台 |
| wiki/主页.md | 编辑 | 导航表新增 5 条目 |
| wiki/术语索引.md | 编辑 | 新增 12 个术语 |

### 预期结果
全量健康检查通过，无断链，反向链接完整。
```

### 5.2 QoderWork Agent -> Human Editor（发现需要人工裁决的冲突）

```markdown
## 简要 Handoff

**发起**: QoderWork Agent | **接收**: Human Editor | **级别**: P1

**已完成**: 全量健康检查发现两个页面内容高度重叠
**待处理**: 决定 [[SEO基因重建批次1]] 和 [[SEO重构工作流v1]] 是否应合并为一个页面，还是保持独立
**涉及页面**: [[SEO基因重建批次1]]、[[SEO重构工作流v1]]
**备注**: 两者都涉及 SEO 重构，前者聚焦具体批次执行结果，后者聚焦方法论。建议保持独立但增加交叉引用。
```

## 六、Handoff 日志归档

### 6.1 日志存储

- 重要的 Handoff 消息（P0/P1 级别）应记录在当日笔记文件中（如 `2026-06-08.md`）
- P2/P3 级别的 Handoff 可在 Agent 会话中保留，不必持久化

### 6.2 日志格式

```markdown
### Handoff 日志 — YYYY-MM-DD

| 时间 | 发起 | 接收 | 级别 | 摘要 | 状态 |
|------|------|------|------|------|------|
| HH:MM | LLM Wiki | QoderWork | P2 | 5 个新页面健康检查 | 已确认 |
| HH:MM | QoderWork | Human | P1 | 命名冲突待裁决 | 待处理 |
```
