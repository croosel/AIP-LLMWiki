# 🚀 最短时间执行路径方法论 (Shortest Path Execution - SPE)

> **核心原则**：以物理结果为导向，优先选择确定性最高、耗时最短、交互环节最少的执行路径。拒绝盲目模拟人类操作。

## 1. 路径路由决策矩阵 (Routing Matrix)

当 Agent 接收到任务时，必须按照以下优先级进行路径选择：

| 优先级 | 路径类型 | 执行方式 | 耗时评价 | 适用场景 |
| :--- | :--- | :--- | :--- | :--- |
| **P0 (极速)** | **API / Plugin 直连** | 调用已连接的官方助手 API 或 Accio 专用 Skill。 | < 2s | 批量上品、调价、数据抓取。 |
| **P1 (高效)** | **Console / JS 注入** | 在浏览器环境中直接操作内存变量或触发前端接口请求。 | 2-5s | 复杂表单填充、绕过繁琐 UI 交互。 |
| **P2 (保底)** | **UI Simulation** | 模拟点击、输入、Snapshot 验证。 | 15-60s | 登录认证、无 API 暴露的非标准页面。 |

## 2. “大脑-手脚”协作模型

- **大脑 (Commander - Accio)**：负责从 **LLM Wikis (Ontology)** 提取真理数据，生成符合品牌要求的 JSON Payload。
- **手脚 (Executor - 生意助手/API)**：负责将 Payload 物理推送到服务器。

## 3. SPE 执行协议 (Protocol)

1.  **[SENSE] 识别目标与工具**：不仅识别任务，还要识别当前环境下已加载的“加速插件”（如生意助手）。
2.  **[REASON] 评估最短路径**：
    - 如果存在 API：禁止使用 UI 点击。
    - 如果存在 JS 注入点：禁止使用 `browser_type` 逐字输入。
3.  **[ACT] 指挥执行**：
    - 将 Wiki 中的变量（如品牌名、SEO 黄金标题）转化为执行参数。
    - 指挥“手脚”完成物理变更。
4.  **[EVAL] 强验证闭环**：
    - 直接读取接口返回结果或回读数据库。只有物理状态变更（如列表中标题已变）才标记为成功。

## 4. 经验集成 (Knowledge Loop)

- **持续优化**：一旦发现某条 P1 路径可以转化为 P0 路径，立即更新此文档。
- **失败沉淀**：如果 P1 路径失败，记录失败特征（如 CSRF Token 缺失），下次任务自动降级到 P2 但保留审计。

## 5. 反自欺协议 (Anti-Illusion Protocol) — 2026-05-24 强化

> **重大教训**：图片替换 API 路径调研后确认 P0 不可达。详见 [LILIS_Execution_PostMortem_ImageReplace_20260524.md §6](LILIS_Execution_PostMortem_ImageReplace_20260524.md)

### 5.1 路径可达性必须经过物理验证

声称"P0 API 路径可用"前，必须满足全部 3 个证据：

| 证据 | 验证方法 | 反例 |
|------|---------|------|
| **权限证据** | `accio-mcp-cli call <tool>` 返回非 `-32002` | -32002 = Agent 类型不允许 |
| **余额证据** | 调用不返回 `-432 Insufficient i-bean balance` | i-bean=0 时所有付费工具均不可用 |
| **物理证据** | 调用后回读对象状态，字段已变更 | 后台提示"成功"不算 |

### 5.2 调研前必须先扫工具池

启动 API 路径前，**先用 3 条命令快速判断可达性**：

```bash
# 1. 工具存在性
accio-mcp-cli keyword <关键词>

# 2. 权限测试（最小参数调用）
accio-mcp-cli call <tool_name> --json '{}'

# 3. 授权状态
accio-mcp-cli call list_all_authorizations --json '{}'
```

任何一个返回 `-32002` / `-432` / `0 个授权` → 立即判定 P0 不可达，跳到 P1/P2。

### 5.3 历史"API 成功"案例需重新审计

⚠️ **2026-05-23 之前 Wiki 中所有"API 路径 <2s 成功"案例需重新审计**：
- 若无 OAuth 授权记录、无 i-bean 余额记录，则历史"API 成功"很可能是浏览器自动化被误标
- 修正方向：标注为 "P2-UI 自动化（被误标为 P0）"

### 5.4 双 Agent 协同的前置条件

"指挥官 Accio + 执行官生意助手"模型成立的前置条件：

1. ✅ 生意助手 Agent 必须有**活跃会话**（否则 `sessions_send` 无法送达）
2. ✅ 生意助手 Agent 必须有 **i-bean 余额**（否则付费工具调不动）
3. ✅ 生意助手 Agent 必须有 **OAuth 授权**（否则 API 直连未建立）

任何一项不满足，"双 Agent 协同"退化为单 Agent UI 自动化。
