# 开发者工具链总览

## 定义

Palantir **Developer Toolchain** 是一套让开发者基于 Ontology 构建新工作流、应用和功能的完整工具体系。开发者可以在自己的开发环境中直接访问 Ontology 的全部能力，从数据集成到应用部署，实现端到端的开发加速。

## 核心组成

```
Foundry Developer Toolchain
├── API & SDK 层
│   ├── Ontology SDK (OSDK) — Python / TypeScript / Java
│   ├── Platform APIs — 平台管理 + 数据操作
│   └── Beta SDKs — Platform Python SDK / TypeScript SDK
├── 开发环境
│   ├── Code Workspaces — JupyterLab / RStudio / VS Code
│   ├── Developer Console — 应用管理 + SDK文档
│   └── Global Branching — 安全隔离开发环境
├── 计算 & 集成
│   ├── Functions — TypeScript v1/v2 / Python / AIP Logic
│   ├── Compute Modules — 交互式容器（任意语言）
│   └── Custom Endpoints — 自定义API端点
└── AI 原生工具
    ├── Palantir MCP — AI构建Foundry应用
    └── Ontology MCP — 外部AI代理访问Ontology

```

## 核心属性

### 1. Ontology SDK (OSDK)

**定位**：开发者访问 Ontology 的主要编程接口。支持 TypeScript、Python、Java 三种语言。

**能力**：
- 读取/查询 Object Types
- 执行 Actions 写入 Ontology
- 调用 Functions
- 运行 AIP Logic 函数（AIP-enabled enrollments）

**生成方式**：Development Console 中创建应用 → 自动生成应用专属 SDK → 包含为你的 Ontology 定制的 API 文档和代码示例

**使用流程**：`Developer Console 创建应用 → 生成 OSDK → 本地 IDE 安装 → OAuth 认证 → 开发`

### 2. Platform APIs

**定位**：编程化管理平台访问和数据源。包括：
- 用户/组/权限管理 API
- 数据集管理 API
- 监控与审计 API
- 数据导入/导出 API

平台 SDK（Beta）：Python SDK 和 TypeScript SDK，与 OSDK 共用客户端但不需要绑定 Ontology。

### 3. Code Workspaces

**定位**：将 JupyterLab / RStudio / VS Code 原生集成到 Foundry。

**关键特性**：
- 单节点运行（非 Spark 集群）
- 内建 Foundry 数据安全、分支管理、构建调度、资源管理
- 适合 ML 模型构建、数据科学、统计分析
- VS Code Workspace 支持快速构建 React 应用（与 Developer Console 集成）
- 可生成 OSDK 直接在 Workspace 中使用

**限制**：单节点不适合大规模数据转换——此时应使用 Pipeline Builder 或 Code Repositories。

### 4. Functions

**定位**：服务器端逻辑执行环境，支持 TypeScript v1/v2、Python、AIP Logic。

**运行模式**：
- **Serverless**（推荐）：按需启动，多版本可并存，安全升级
- **Deployed**：专用资源常驻，适用于特殊场景

**核心使用场景**：Workshop 数据展示、派生计算、复杂 Ontology 编辑、外部 API 调用、Pipeline Builder 集成。

### 5. Compute Modules

**定位**：在 Foundry 中部署交互式容器，可使用任意语言运行现有代码。

**场景**：将第三方 ML 模型带入平台，不需重写即可集成到 Foundry 工作流。

### 6. Custom Endpoints

**定位**（Beta）：自定义 API 端点，匹配企业现有标准。

**能力**：定义自定义 URL 模式、请求/响应形状 → 底层由 Ontology Actions/Functions 驱动。

### 7. Palantir MCP

**定位**：Model Context Protocol 实现，让 AI IDE 和 AI Agent 自动构建 Foundry 应用。

**两大 MCP 方案**：

| MCP 类型 | 定位 | 能力 |
|---------|------|------|
| **Palantir MCP** | Ontology 构建者 | 修改 Ontology 类型，不可写数据 |
| **Ontology MCP** | Ontology 消费者 | 暴露 Object Types/Actions/Functions 为 MCP Tools，外部 AI 可读写数据 |


## 业务用途

### 开发者工具链的定位

开发者工具链是 Foundry 从"平台工具"到"平台生态"的关键转变。它不仅让 Palantir 内部开发者构建应用，更让**客户自己的开发团队**能够在 Palantir 平台上构建定制化解决方案。

### Global Branching

全域分支机制为开发创造安全隔离环境——开发者在 Branch 中测试完整工作流，验证通过后合并到生产。

### 典型开发流程

```mermaid
Developer Console 创建应用 → 生成 OSDK → VS Code/Code Workspaces 开发
                                          ↓
                Functions 编写业务逻辑 → Workshop 消费 Function
                                          ↓
                Custom Endpoints 暴露 API → 外部系统集成
```

### LILIS 开发者工具链映射

由于 LILIS 没有 Foundry 实例，以下是等效工具映射和建设路径：

| Foundry 工具 | LILIS 等效方案 | 使用场景 |
|-------------|--------------|---------|
| **OSDK (TypeScript)** | Alibaba API Client (封装) | 程序化读写产品/P4P/订单数据 |
| **OSDK (Python)** | Python Pandas + Alibaba API | 数据分析 + 自动化脚本 |
| **Functions** | Google Cloud Functions / AWS Lambda | P4P定时调价、询盘自动分类 |
| **Code Workspaces** | Jupyter Notebook (本地) | P4P数据分析、竞品研究 |
| **Custom Endpoints** | Flask/FastAPI 封装 Alibaba API | 为运营仪表盘提供标准化 REST API |
| **Developer Console** | Git + CI/CD Pipeline | 版本管理 + 自动化部署 |
| **Compute Modules** | Docker Containers | 部署 ML 模型（询盘分类器等） |
| **Palantir MCP** | Accio Work Agent Skills | AI 代理辅助运营操作 |

### 分阶段建设建议

**Phase 1 — API 封装层**：
```python
# LILIS OSDK 等效：标准化 Alibaba API 调用
from lilis_osdk import Product, P4P_Campaign, Inquiry

# 查询所有产品的 P4P 表现
products = Product.where(status="active")
for p in products:
    keywords = p.get_p4p_keywords()
    ctr_trend = p.get_ctr_trend(days=30)
```

**Phase 2 — Serverless Functions**：
- P4P Budget Alert Function：日预算消耗>80% → 通知 → 自动调整
- Inquiry Classifier Function：询盘内容 → AI分类 → 标记优先级
- Product Health Score Function：多维度计算产品健康度

**Phase 3 — MCP-Style Agent**：
- Accio Work Agent 通过 MCP 协议暴露 LILIS 运营工具
- AI 代理可直接操作：`查询预算 → 分析趋势 → 调整出价`

## 常见问题

**Q: OSDK 和 Platform APIs 的区别？**
A: OSDK 是面向 Ontology 的高层 SDK（对象读写/动作/函数），Platform APIs 是底层平台管理 API（用户/权限/数据集管理）。OSDK 更易用，Platform APIs 更灵活。

**Q: Serverless Functions 和 Deployed Functions 如何选择？**
A: 默认用 Serverless（多版本共存、按需启动、自动扩展）。仅在需要专用资源池、极低延迟或特殊网络配置时才用 Deployed。

**Q: Code Workspaces 能替代 Pipeline Builder 吗？**
A: 不能。Code Workspaces 在单节点运行，Pipeline Builder 基于 Spark 集群。大规模数据转换应使用 Pipeline Builder。

**Q: LILIS 没有 Foundry 实例，能用 Palantir MCP 吗？**
A: Palantir MCP 需要 Foundry 实例。但 Accio Work 的 Agent + Skill 体系可构建等效的 AI 代理开发能力。
