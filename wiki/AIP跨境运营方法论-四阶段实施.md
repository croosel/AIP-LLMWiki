# AIP 跨境运营方法论：四阶段实施

> **核心理念**: 这不是学术框架——这是一个卖家拿到手就能一步步照做的运营手册
>
> **设计原则**: 每一步都对应可量化的商业回报，不做解释不了 ROI 的事情
> **覆盖目标**: 60% 国际站运营痛点。剩下的 40% 是 Phase 3/4 的事

---

## 方法论总览

```
阶段1: 数据止血 (Week 1-2)      →  省时 + 了解全貌
阶段2: 诊断增效 (Week 3-4)      →  省钱 (P4P) + 修复风险
阶段3: 半自动运营 (Month 2-3)   →  省人 + 提高转化
阶段4: AI 托管 (Month 4+)       →  替代运营团队

每一阶段的 ROI 都可量化，不达标不进下一阶段。
```

---

## 阶段 1：数据止血（Data Foundation）

> **目标**: 把散落在阿里后台各处的数据汇入 LLM Wiki，建立完整数据底座
> **时间**: 1-2 周
> **商业价值**: 卖家首次看清店铺全貌，每天节省 30+ 分钟手动查数据

### 步骤 1.1：搭建 Ontology 数据层（Day 1-3）

**做什么**：

在 Accio Work 的 LLM Wiki 下创建以下 6 个数据层页面：

```
LLM Wiki/店铺数据/
├── Store-Overview.md       ← 店铺总览
├── Product-List.md         ← 全量产品表
├── P4P-Campaigns.md        ← 广告详情
├── Inquiry-Log.md          ← 询盘记录
├── Compliance-Status.md    ← 合规状态
└── Diagnostic-History.md   ← 诊断记录（初始化为空）
```

**每个页面的表结构模板**（以 Product-List.md 为例）：

```markdown
# 产品列表 | 更新: {{today}}

| ID | 产品名 | 星级 | 价格 | 30天曝光 | 30天点击 | 30天询盘 | 转化率 | P4P花费 | P4P询盘 | 状态 |
|----|--------|------|------|---------|---------|---------|--------|---------|---------|------|
| 1  | ...    | ...  | ...  | ...     | ...     | ...     | ...    | ...     | ...     | ...  |
```

**为什么这样做**: 标准化的表结构 = 轻量级 Ontology。Agent 可以用 `read` 工具精确查询特定行和列，不需要搜索整个 Wiki。

**成功的标志**: 6 个页面创建完毕，Agent 能通过 `[[Product-List]]` 链接定位到具体产品。

参考: [[LLM-Wiki统一知识底座设计#三、数据层联动机制]]、[[LILIS数据模型v2]]

### 步骤 1.2：打通数据管道（Day 4-7）

**做什么**：

设置 Cron 定时任务，自动从阿里国际站 MCP 拉取数据并写入 Wiki：

```
Cron Job 1: 每日 06:00 拉取店铺总览 → 更新 Store-Overview.md
Cron Job 2: 每日 06:30 拉取产品列表 → 更新 Product-List.md
Cron Job 3: 每日 07:00 拉取 P4P 广告数据 → 更新 P4P-Campaigns.md
Cron Job 4: 每日 07:30 拉取询盘记录 → 更新 Inquiry-Log.md
Cron Job 5: 每周一 08:00 扫描合规项 → 更新 Compliance-Status.md
```

**技术实现**（Accio Work 现有工具）：

1. 使用 `cron` 工具设置定时任务
2. 使用 `alibaba-*` MCP 工具拉取数据
3. Agent 用 `write` 工具写入对应 Wiki 页面

**为什么这样做**: Cron 自动化是 AIP Logic 的低成本替代——不需要等平台原生支持 Logic，用现有工具就能实现「定时数据管道」。

**成功的标志**: 连续 3 天数据自动更新无中断，Wiki 数据层与阿里后台数据一致。

参考: [[LLM-Wiki统一知识底座设计#四、Wiki → Ontology 映射]]

### 步骤 1.3：创建诊断 Skill（Day 8-10）

**做什么**：

在 Agent 下创建 8 个核心 Skill（Skill.md），每个 Skill 配备清晰的触发语、数据来源和输出格式。

| Skill | 触发词 | 数据来源 | 输出格式 |
|-------|--------|---------|---------|
| **p4p-diagnose** | "检查 P4P" / "广告有没有浪费" | P4P-Campaigns.md + Product-List.md | 浪费词清单 + 建议出价 |
| **store-health** | "店铺健康" / "最近有什么问题" | Store-Overview.md + [[商家星等级]] | 问题列表 + 影响评估 |
| **product-audit** | "检查产品" / "哪些产品要优化" | Product-List.md + [[商品等级体系]] | 低质品清单 + 晋级方案 |
| **listing-check** | "检查 Listing" / "这个产品怎么改" | Product-List + [[标题优化黄金公式]] | 修改建议清单 |
| **rule-qa** | "XX规则是什么" | 公共规则层 Wiki 页面 | 规则解释 + 引用 |
| **compliance-scan** | "合规检查" / "有没有风险" | Compliance-Status.md + [[实战避坑指南]] | 风险清单 + 倒计时 |
| **quick-compare** | "该不该提价" / "降价有用吗" | Product-List + [[Alibaba排序逻辑]] | 决策建议 + 数据依据 |
| **daily-push** | 自动触发（Cron） | Store-Overview.md | 每日运营简报 |

**每个 Skill.md 的标准模板**：

```markdown
# Skill: p4p-diagnose
## 触发
当用户询问 P4P 广告是否有浪费 / ROI 是否正常 / 该不该调整出价时触发。

## 执行流程
1. read P4P-Campaigns.md → 获取所有广告计划数据
2. 对每个关键词计算：ROI = 询盘数 / 花费
3. 标记 ROI = 0 且花费 > $50 的关键词
4. 读取 [[P4P投流完整方法论]] → 获取同行出价参考
5. 输出诊断卡片

## 输出格式
- 浪费关键词清单（词 + 花费 + 零转化天数）
- 建议出价（当前 vs 建议 vs 同行均价）
- 预估节省金额
- 引用来源（P4P-Campaigns.md 第 X 行 + [[P4P投流完整方法论]] §X）
```

**为什么这样做**: 标准化的 Skill 模板确保 Agent 每次回答都走同样的推理路径，输出格式一致——这是「可预期的 AI」，不是「猜运气的 AI」。

**成功的标志**: 每个 Skill 能在 1 次对话内给出带数据引用的具体建议。

参考: [[Skills技能集与60-40覆盖]]、[[AI-FDE完整架构#Skills]]

| 指标 | 目标 | 测量 |
|------|------|------|

## 阶段 2：诊断增效（Diagnostic Intelligence）

> **目标**: 5 个 AIP Logic 诊断管道上线，Agent 主动发现 + 推送问题
> **时间**: 2-3 周
> **商业价值**: P4P 直接省钱 $200-800/月，低质品修复避免降权

### 步骤 2.1：上线 AIP Logic 诊断管道（Day 1-10）

参照 [[AIP跨境运营中枢-商业系统架构#AIP Logic 管道设计|系统架构中的 5 个管道]]，用 Cron + Agent 指令实现：

**管道 1: P4P 浪费检测（最先上线——离钱最近）**

```
Cron: 每日 08:00 触发
Agent 指令:
  1. read Product-List.md → 获取所有产品
  2. read P4P-Campaigns.md → 获取所有广告数据
  3. 对于每个关键词：计算 ROI = 询盘数 / 花费
  4. 筛选条件：ROI = 0 AND 花费 > $50 AND 连续 ≥ 7 天
  5. write Daily-Alert.md → 追加今日发现
  6. 生成诊断卡片 → 推送通知
```

**管道 2: 星等级诊断**

```
Cron: 每日 08:30 触发（在管道 1 之后）
Agent 指令:
  1. read Store-Overview.md → 获取最新数据
  2. 对比 7 天前数据 → 计算各因子变化
  3. 定位下降因子 → 读取 [[商家星等级]] 规则 → 给出修复路径
  4. 如果总分下降 > 5% → write Daily-Alert.md + 推送紧急告警
```

**管道 3-5**: 低质品扫描 / 合规风险扫描 / 转化异常检测（按优先级依次上线）

**为什么这样做**: 管道是从「被动问答」到「主动保镖」的质变——卖家不需要每天记得问，Agent 自动发现问题并通知。

**成功的标志**: 连续 7 天管道正常运行。至少 1 次管道发现了卖家自己没发现的问题。

参考: [[AIP-Logic核心概念#Block 链式组合]]、[[Sense-Reason-Act架构#Reason Layer]]

### 步骤 2.2：植入 AIP Evals 质量反馈（Day 11-14）

**做什么**：

在每次诊断输出后，Agent 自动写入质量追踪记录到 Diagnostic-History.md：

```markdown
## 诊断记录 #047 | 2026-05-25
- 触发: Cron 管道 1 (P4P 浪费检测)
- 发现: 3 个关键词零转化
- 建议: 暂停 "wedding arch"、"party tent"、"flower wall"
- 卖家操作: 已暂停 2 个 / 保留 1 个
- 30 天验证: 待追踪
- 准确度自评: ⭐⭐⭐⭐ (2/3 被采纳)
```

**为什么这样做**: Evals 是产品「自我进化」的引擎。每 100 次诊断 → 自动识别低准确率规则类型 → 反馈到 Wiki 规则层修正。没有 Evals 的 AI 产品是「猜运气的工具」。

**成功的标志**: Diagnostic-History.md 累计 50+ 条记录，其中 80%+ 有卖家操作反馈。

参考: [[AIP-Evals]]、[[AIP评估套件深度]]

### 步骤 2.3：启动内测（Day 15-21）

**做什么**：

- 邀请 3-5 个种子卖家（优先从已知团队：Kenny/Cola）
- 免费试用 14 天
- 重点收集：
  - 诊断建议准确率（卖家采纳比例）
  - 卖家追问次数（0 次追问 = 建议足够完整）
  - 卖家自述的 P4P 节省金额
  - "你最想 Agent 还能做什么？"

**为什么这样做**: 内测不是验证「技术是否 work」——是验证「卖家是否愿意为这个付费」。

**成功的标志**: 3 个卖家中有 2 个表示「愿意继续用，愿意付 $29/月」。

## 阶段 3：半自动运营（Semi-Automation）

> **目标**: Agent 从「告诉你问题」升级为「帮你解决问题」
> **时间**: 1-2 个月
> **商业价值**: 时间节省从 30 分钟/天提升到 90+ 分钟/天。卖家只需「确认」操作

### 步骤 3.1：实现 Staged Act（一键批准）（Week 1-4）

**做什么**：

对照 [[动力层与写回#三层执行模式]]，实现操作确认 → 自动执行 → 审计记录：

```
Agent 输出诊断:
  ⚠️ 发现 3 个零转化关键词
  ┌─────────────────────────────────────┐
  │ ☐ "wedding arch" → 花费 $120 → 暂停 │
  │ ☐ "party tent"   → 花费 $85  → 暂停 │
  │ ☐ "flower wall"  → 花费 $67  → 暂停 │
  ├─────────────────────────────────────┤
  │ 预估节省: $272/月                    │
  │ [全部暂停] [逐项确认] [忽略]           │
  └─────────────────────────────────────┘

卖家点击 [全部暂停] → Agent 调用 alibaba MCP 工具执行 → 写入 Audit-Log.md
```

**实现**：

1. 在 Skill 输出中加入操作确认选项
2. 卖家回复确认 → Agent 调用 `alibaba-*` MCP 工具的写操作
3. 每次执行后写入 Audit-Log.md（记录时间、操作、结果、回滚方式）

**成功的标志**: Agent 至少执行 10 次操作，零次出错，卖家零次回滚。

参考: [[AI-FDE完整架构#Staged Mode]]、[[动力层与写回]]

### 步骤 3.2：诊断卡片病毒传播（Week 3-6）

**做什么**：

每次诊断自动生成可分享的「诊断卡片」，嵌入分享按钮：

```
┌─────────────────────────────────────┐
│  🔍 AI 运营诊断                       │
│  📅 May 25, 2026                     │
│  ⚠️ 发现 3 个零转化 P4P 关键词          │
│  💰 预估节省: $272/月                 │
│  ⏱️ 诊断用时: 2 分钟                  │
│  🤖 来自 AI 运营顾问                   │
│  [查看详情]  [分享到卖家群]              │
└─────────────────────────────────────┘
```

**为什么这样做**: 每张卡片 = 一个自带社交证明的产品广告。卖家分享的意愿远高于「转发一个产品链接」。

**成功的标志**: 10% 的用户产生过至少 1 次分享行为，1 次分享带来了至少 1 个新增注册。

参考: [[商业化策略与病毒增长#五、病毒式传播飞轮]]

### 步骤 3.3：付费墙 + 定价上线（Week 5-8）

**做什么**：

- 基础版 $29/月（诊断 + 建议）→ 占 60% 用户
- 高级版 $99/月（一键执行 + 每日推送 + 周报）→ 占 30% 用户
- 定制版 $299/月（多店铺 + 专属 Agent）→ 占 10% 用户

**成功的标志**: 月收入 > $2,000，月留存 > 80%。

参考: [[商业化策略与病毒增长#一、商业模式]]

| 指标 | 目标 |

| 月付费用户 | > 50 |

## 阶段 4：AI 托管运营（Autonomous Operations）

> **目标**: Agent 全权负责 P4P 调价、低质品管理、合规监控。卖家只看周报
> **时间**: Month 4+
> **商业价值**: 一个月费顶一个运营的年薪。且不会离职

详细路线见 [[三阶段升级路径-Sense-Reason-Act#Phase 3]]。

**进入阶段 4 的硬条件**：
- Staged Act 批准率 > 90%
- 零回滚事件 ≥ 60 天
- Evals 准确率 > 90%
- 月收入 > $10,000

## 五、为什么这个方法论「可实施」

| 特征 | 说明 |
|------|------|
| **不依赖平台原生功能** | AIP Logic / Ontology / Evals 的核心思想用 Cron + Wiki + Agent SOP 即可实现 |
| **每一步有成功标准** | 不达标不进下一阶段，避免在错误方向上投入 |
| **商业价值可量化** | P4P 节省 / 时间节省 / 星等级改善 → 直接对应卖家收入 |
| **离钱近的优先做** | 阶段 1-2 专注 P4P 浪费检测（直接省钱），不做创意生成等「锦上添花」功能 |
| **经验累积形成壁垒** | 每多一个客户 → Ontology 模式库更完善 → 诊断更精准 → 壁垒增高 |

## 六、从卖家视角看方法论

```
Week 1:  "这个 Agent 能帮我看数据？行，试试。"
           ↑ Agent 连接店铺，自动建立 Wiki 数据层

Week 2:  "它居然发现了我没发现的 P4P 浪费！省了 $300！"
           ↑ P4P 浪费检测管道第一次推送

Week 3:  "我今天没检查店铺，Agent 主动告诉我星等级在掉。"
           ↑ 从被动问答变成主动保镖

Week 4:  "我直接点了确认，它就帮我把词停了。只用 30 秒。"
           ↑ Staged Act 首次使用

Month 2: "我把 Agent 推荐给了群里另外 3 个卖家。"
           ↑ 病毒传播启动

Month 4: "我现在每周只看 Agent 的周报，P4P 让它自己管。"
           ↑ 全托管运营
```
