# 链接类型深度

## 定义

**Link Types** 是 Ontology 中两个 Object Type 之间关系的 Schema 定义。单个链接实例称为 **Link**。Link Types 是 Ontology 关系建模的核心，决定了数据结构如何关联、导航和分析。

## 核心属性

### 基数类型（Cardinality）

| 基数 | 实现方式 | 使用场景 | 限制 |
|------|---------|---------|------|
| **一对一** | Foreign Key | 一个员工对应一个工位 | 需要唯一性约束 |
| **一对多** | Foreign Key (多方持有一方PK) | 一个部门对应多个员工 | 最常用 |
| **多对多** | Join Table (两方PK对表) | 一个产品对应多个供应商 | 需要额外数据集 |

**Foreign Key 方式**：Object Type A 的某属性存储 Object Type B 的 Primary Key——适用于一对一和一对多。

**Join Table 方式**：单独的中间表存储双方 Primary Key 对——多对多必须使用此方式。Object-backed Link Types 也是基于此模式。

### 创建流程

Ontology Manager 提供引导式创建助手：
1. 选择关系类型（Foreign Key 或 Join Table）
2. 选择两端的 Object Type
3. 指定 Primary Key 映射
4. 命名 Link Type（API Name）

半途退出助手后可手动补全 Key 和 API Name 配置。

### Interface Link Type 约束

Interface 可以定义 Link Type 约束：
- 指定实现此 Interface 的 Object Type 必须拥有到特定目标类型的链接
- 支持 ONE 或 MANY 基数（类似一对一/一对多）
- 让跨类型的关系导航具有类型安全保障

### 衍生属性（Derived Properties）

Derived Properties 不存储数据，而是通过 Link Type 链从关联对象中拉取信息，并可应用聚合操作。

**价值**：避免数据冗余。例如 `Department` 的 "Total Salary" 不需要存储，可以通过 `Department → Employee` 链接聚合 `Employee.Salary` 动态计算。

**聚合类型**：
- COUNT、SUM、AVG、MIN、MAX
- 支持过滤条件（如只计算 Active Employees）

**多跳衍生（Multi-hop）**：
支持最多 3 级链式导航：
```
Department → Employee → Project → Client
Department.Project_Names = 聚合(Project.Name)
```

**已知限制**：
- 不支持 OSv1 对象类型
- 不能用于文本搜索/关键词过滤
- 当前 TypeScript OSDK 不支持包含 Struct 属性的查询
- 不能与 Inline Actions 共存
- 不能作为 Primary Key
- 不支持 Default Ontology 中的 Ontology Condition

### 元数据类型类（Metadata Type Classes）

Type Classes 是为 Properties / Link Types / Action Types 添加的语义元数据标注。不同应用解释这些标注来改变渲染行为。

**hubble 类型类**（对象展示）：
| Type Class | 效果 |
|-----------|------|
| `media_url` | 在 Object View 中渲染为媒体项 |
| `icon` | URL 属性作为对象图标显示 |
| `hierarchy.parent` | 显示层级面包屑（如"欧洲→法国→巴黎"） |

**timeseries 类型类**（时序数据）：
| Type Class | 效果 |
|-----------|------|
| `timeseries_is_deprecated` | 过滤废弃时序数据 |
| `timeseries_is_value_inverted` | Y轴反转（用于深度/井下数据） |
| `parent` | 描述时序与父级关系 |

**schedules 类型类**（调度）：
| Type Class | 效果 |
|-----------|------|
| `schedulable-start-time` | 标记调度开始时间属性 |
| `schedulable-end-time` | 标记调度结束时间属性 |
| `segment-by` | 启用调度分段着色 |

**analyzer 类型类**：影响索引行为（非展示用途）。

## 业务用途

### 为什么 Link Types 是 Ontology 的命脉

数据只有在关联时才产生真正的价值：
- 孤立的产品数据 → 信息量 1x
- 产品 + 关联的 P4P 关键词 → 信息量 5x
- 产品 + P4P + 询盘 + 订单 + 供应商 → 信息量 50x

Link Types 将这 50x 的信息关联变成了可系统化导航的结构，而非一次性手工查询。

### Ontology 迁移

资源可在不同 Ontology 之间迁移（Ontology Manager → Migrate resources）。迁移改变资源权限但不改变底层数据源权限。**约束**：不能将私有 Ontology 的 Object Type 迁移到 Default Ontology（除非最初创建在 Default 中）。迁移必须一起迁移关联资源，否则失败。

## 外贸应用

### LILIS Link Type 设计

```
Product （产品）
  ├── has_keyword → P4P_Keyword （多对多，Join Table）
  ├── receives → Inquiry （一对多，FK: Inquiry.product_id）
  ├── belongs_to → Category （多对一，FK: Product.category_id）
  └── supplied_by → Supplier （多对一，FK: Product.supplier_id）

Inquiry （询盘）
  ├── converts_to → Order （一对一，FK）
  └── from_buyer → Buyer （多对一，FK）
```

### 关键设计决策

1. **Product ↔ P4P_Keyword** 必须用多对多：一个产品投多个关键词，一个关键词被多个产品使用
2. **Inquiry → Product** 用一对多：一条询盘绑定一个产品
3. **Derived Property 应用**：`Product.Total_Inquiries` = COUNT(Inquiry where product_id = this.id)、`Product.Avg_CTR` = AVG(P4P_Keyword.CTR where product_id = this.id)
4. **Metadata Type Class**：给 `Product.main_image_url` 添加 `media_url` 类型类 → 在视图中自动渲染为图片

## 常见问题

**Q: Foreign Key 和 Join Table 如何选择？**
A: 一对一和一对多用 Foreign Key（简单、高效）；多对多用 Join Table（必须）。Join Table 可用于任何基数但复杂度更高。

**Q: Derived Properties 和 Functions (FOO) 的区别？**
A: Derived Properties 是 Schema 级别的声明式聚合（通过 Link Type 自动计算），FOO 是编程式的计算函数。Derived Properties 更简单但能力有限，FOO 更灵活但需开发。

**Q: Type Classes 和 Properties 本身的区别？**
A: Properties 定义数据字段；Type Classes 是附加在 Properties 上的语义标注，告诉应用"这个属性应该怎么展示/使用"。同一个 Property 可以有多个 Type Class。
