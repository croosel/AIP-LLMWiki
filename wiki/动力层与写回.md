# 动力层与写回机制

## 定义
AIP 的**动力层 (Kinetic Layer)** 是实现「数字决策 → 物理世界变更」的桥梁。它通过**写回 (Write-back)** 机制将 AI 的推理结果同步到外部生产系统（如 SAP、ERP、Alibaba MA），并通过**预提交 (Pre-commit)** 验证确保事务一致性，完成从「记录系统」到「行动系统」的跨越。

## 核心属性
| 属性 | 值 |
|------|-----|
| 层级 | 本体架构的动力学层 |
| 核心机制 | Write-back + Pre-commit |
| 外部目标 | SAP / ERP / CRM / MA |
| 通信方式 | Webhooks |
| 原子性保障 | 预提交验证 → 外部确认 → 本体更新 |
| 转型方向 | System of Record → System of Action |

## 业务用途

### 写回 (Write-back) 流程
```
AI 推理出决策
  → 调用 Apply Action
  → 生成写回指令 (Webhook)
  → 发送到外部系统 (SAP/ERP/Alibaba)
  → 外部系统执行并返回结果
  → 本体数字孪生状态更新
```

### 预提交 (Pre-commit) 验证
- **双重确认**：先向外部系统发起请求，确认成功后本体才更新
- **防止脱节**：避免本体数据与实际系统状态不一致
- **回滚保障**：外部执行失败 → 本体不更新 → 保持一致性

### 事务一致性保障
| 步骤 | 操作 | 失败处理 |
|------|------|---------|
| 1 | AI 生成修改指令 | 逻辑验证（Evals） |
| 2 | Action 规则校验 | 拦截（如库存不能为负） |
| 3 | 发送写回请求 | 重试或终止 |
| 4 | 外部系统执行 | 超时回滚 |
| 5 | 本体状态同步 | 仅成功时更新 |

## 外贸应用
> 参见 [[AIP神经符号架构]] · [[AIP-Logic]] · [[P4P投流策略]]

对 LILIS 的映射场景：
- **P4P 出价调整**：AI 分析 ROI → 决策「关键词 X 降低出价 20%」→ 动力层通过 Alibaba MA API 自动执行
- **库存同步**：询盘预订 500 件 → 动力层写回 ERP 库存系统 → 预留库存
- **产品发布**：AI 生成新品 → 动力层调用 Alibaba 发品 API → 创建商品

## 常见问题
**Q: 动力层与「应用行动 (Apply Action)」的关系？**
A: Apply Action 是 AIP Logic 中触发的操作指令，动力层是这个指令的执行通道——负责将 Action 翻译为具体的外部系统调用。

**Q: 如果外部系统挂了，写回会怎样？**
A: 预提交机制保障：外部系统故障 → 事务失败 → 本体不更新 → AI Agent 收到失败通知，可选择重试或人工干预。
