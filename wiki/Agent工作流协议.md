# Agent 工作流协议

> 本协议定义了所有 AI Agent 在操作 AIP-LLMWiki 知识库时必须遵守的统一规范。  
> 无论是 [[LLM-Wiki统一知识底座设计|LLM Wiki Agent]]、QoderWork Agent、Cursor、Claude 还是其他任何 Agent，均受本协议约束。  
> 本协议的权威版本存放于 `wiki/Agent工作流协议.md`，由 Human Editor 审批后生效。

---

## 一、知识库边界定义

### 1.1 目录结构契约

```
AIP-LLMWiki/
├── wiki/           ← 写入区：所有结构化知识页面
├── raw/            ← 只读区：原始资料，禁止任何 Agent 修改
├── schema/         ← 规则区：由 schema-agent 专管
├── *.md            ← 根目录：欢迎页、每日笔记等入口文件
├── *.canvas        ← 画布文件：Obsidian Canvas
└── *.base          ← 数据库文件：Obsidian Bases
```

| 区域 | 路径 | 权限 | 说明 |
|------|------|------|------|
| **写入区** | `wiki/` | 所有 Agent 可创建/编辑 | 结构化知识页面的唯一存放位置 |
| **只读区** | `raw/` | **所有 Agent 禁止修改** | 原始资料归档，仅作为信息源引用 |
| **规则区** | `schema/` | 仅 schema-agent 可编辑 | 规则与标准的唯一定义处 |
| **根目录** | `/` | 谨慎编辑 | 欢迎页、每日笔记、Canvas 等 |

### 1.2 只读区规则（raw/）

- **绝对禁止**：任何 Agent 不得对 `raw/` 目录下的文件执行写入、删除、重命名、移动操作
- **引用方式**：通过 `source` frontmatter 字段引用，如 `source: "[[Palantir_AIP_Wiki]]"`
- **新增资料**：仅 Human Editor 可向 `raw/` 添加新的原始资料
- **例外情况**：若确实需要在 `raw/` 中创建文件（如 Agent 执行日志），须获得 Human Editor 明确授权

### 1.3 规则区规则（schema/）

- `schema/规则.md.md` 是知识库规则与标签体系的**唯一权威来源**
- 术语定义库、标签规范、页面结构模板均以 `schema/规则.md.md` 为准
- 当 wiki 页面内容与 schema 规则冲突时，**schema 优先**
- Agent 发现 schema 规则需要更新时，应在页面的 `## 相关链接` 处标注，并通知 Human Editor

### 1.4 写入区规则（wiki/）

- 所有 Agent 生成的知识页面**必须**存放于 `wiki/` 目录
- 页面文件名为 `.md` 格式
- 当前 wiki/ 目录下共有约 143 个页面，Agent 应熟悉现有页面避免重复创建

## 二、页面创建规范

### 2.1 YAML Frontmatter 要求

每个新建 wiki 页面**必须**包含以下 YAML frontmatter：

```yaml
---
tags:
  - 标签1
  - 标签2
created: YYYY-MM-DD
source: "来源说明（引用 raw/ 文件或其他来源）"
---
```

**字段说明**：

| 字段 | 必填 | 格式 | 说明 |
|------|------|------|------|
| `tags` | 是 | YAML 列表 | 至少 1 个标签，须符合标签分类体系（见 2.6） |
| `created` | 是 | `YYYY-MM-DD` | 页面创建日期 |
| `source` | 否 | 字符串 | 来源资料引用，可用 wikilink 格式 |
| `verified` | 否 | `true`/`false` | 内容是否经人工审核验证 |
| `priority` | 否 | 数字 | 优先级（1=最高） |

### 2.2 页面结构模板

标准知识页面应包含以下章节（按需裁剪，但**定义**和**相关链接**为必须章节）：

```markdown
# 页面标题

## 定义
（术语/概念的清晰定义，一段话概括核心）

## 核心属性
（关键属性、参数、分类，通常用表格呈现）

## 业务用途
（在企业运营中的价值和应用场景）

## 外贸应用
（在 Alibaba/LILIS 场景中的具体应用，含实例）

## 常见问题
（海外客户高频问题 & 标准答案）

## 相关链接
- [[相关页面1]]
- [[相关页面2]]
```

**Stub 页面**（自动创建的占位页面）使用以下格式：

```markdown
---
tags:
  - stub
  - 待完善
  - 相关标签
created: YYYY-MM-DD
---

# 页面标题

> 本页面为自动创建的 stub，尚待 Agent 补充完整内容。

## 定义

（待补充）

## 相关链接

- [[相关页面1]]
- [[相关页面2]]
```

### 2.3 命名规范

| 类别 | 命名语言 | 示例 |
|------|----------|------|
| LILIS 业务 / 外贸运营 | **中文优先** | `P4P投流策略.md`、`商品等级体系.md`、`转化率优化体系.md` |
| Palantir / 技术概念 | **英文或中英混合** | `AIP-Logic.md`、`Ontology.md`、`Functions运行时.md` |
| 混合主题 | **中英混合** | `LLM-Wiki统一知识底座设计.md`、`AIP跨境运营中枢-商业系统架构.md` |
| 复盘 / 报告 | **英文或中英混合** | `Weekly_Review_协议.md`、`LILIS_AIP_Failure_PostMortem_20260523.md` |

**命名规则**：

- 文件名不含空格，用连字符 `-` 或下划线 `_` 连接
- 中文页面名不使用空格，直接连接（如 `商品等级体系`）
- 避免特殊字符（`/`、`\`、`:`、`?`、`*`、`"`、`<`、`>`、`|`）
- 创建前**必须**检查是否已存在同名或相似名称的页面

### 2.4 Wikilink 格式规范

| 规则 | 正确 | 错误 |
|------|------|------|
| 不加 `.md` 后缀 | `[[AIP总览]]` | `[[AIP总览.md]]` |
| 不使用反斜杠转义 | `[[Ontology深度总览]]` | `[[Ontology深度总览\]]` |
| 用管道符设置显示文字 | `[[规则\|schema/规则]]` | `[[schema/规则.md.md]]` |
| 跨目录引用加路径 | `[[schema/规则\|规则]]` | `[[../schema/规则.md.md]]` |
| 别名页面指向主页面 | `[[Weekly_Review_协议]]` | 直接复制内容 |

**管道符别名使用场景**：

- 引用 schema 目录文件：`[[schema/规则|规则]]`
- 显示文字与文件名不同：`[[P4P投流完整方法论|P4P 关键词]]`
- 嵌入上下文语境：`[[标题优化黄金公式|标题]]不够精准`

### 2.5 重定向/别名页面规范

当同一概念有多个名称时，创建重定向页面（参考 [[Weekly Review协议]] 和 [[规则]] 的做法）：

```markdown
---
tags:
  - stub
  - 待完善
  - 相关标签
created: YYYY-MM-DD
---

# 别名页面标题

> 本页面为别名/重定向页面。完整内容请参见 [[主页面名称]]。

本页面是 [[主页面名称]] 的别名，用于兼容不同引用格式。

## 相关链接

- [[主页面名称]]
```

### 2.6 标签分类体系

标签以 `schema/规则.md.md` 为权威来源。当前标准标签集：

**领域标签**：

| 标签 | 适用范围 |
|------|----------|
| `Palantir` | Palantir 平台相关 |
| `AIP` | AIP 功能相关 |
| `LILIS` | LILIS 品牌相关 |
| `Alibaba国际站` | Alibaba 平台运营相关 |
| `外贸跨境` | 外贸/跨境通用 |
| `婚庆用品` | 婚庆行业相关 |

**内容类型标签**：

| 标签 | 适用范围 |
|------|----------|
| `概念` | 基础概念定义 |
| `SOP` | 标准操作流程 |
| `方法论` | 方法论与框架 |
| `架构` | 系统架构与设计 |
| `数据分析` | 数据分析相关 |
| `工作流` | 工作流与自动化 |
| `客户问答` | 含 FAQ 内容 |

**管理标签**：

| 标签 | 适用范围 |
|------|----------|
| `stub` | 占位页面，待补充内容 |
| `待完善` | 内容不完整，需要扩充 |
| `元页面` | 关于知识库自身的元页面 |
| `知识库治理` | 治理规则与协议 |

**新增标签流程**：Agent 如需新增标签，须在本页面的 `## 相关链接` 处标注建议，并通知 Human Editor 审批后更新 `schema/规则.md.md`。

## 三、双向链接协议

### 3.1 正向链接规则（新页面 -> 已有页面）

- 每个新建页面**必须**在 `## 相关链接` 中链接至少 **2 个**已有的相关页面
- 页面正文中应自然嵌入 `[[wikilinks]]`，而非仅在末尾堆砌链接
- 优先链接到概念的上位页面或强关联页面

### 3.2 反向链接规则（已有页面 <- 新页面）

当创建页面 A 并在其中引用了页面 B 时：

1. 检查页面 B 的 `## 相关链接` 部分
2. 判断页面 A 是否与页面 B 有足够的关联性
3. 如果关联性强（同领域、有直接数据依赖、互为因果），则在页面 B 的 `## 相关链接` 中添加 `[[页面A]]`
4. 如果关联性弱（仅间接引用、跨领域弱关联），则不添加反向链接

**判断标准**：
- 强关联：同一业务链条（如 [[P4P投流策略]] <-> [[P4P投流完整方法论]]）
- 强关联：数据依赖关系（如 [[LILIS数据模型v2]] <-> [[Store-Overview]]）
- 弱关联：仅概念相似但无直接业务联系

### 3.3 Hub 页面例外

以下 Hub 页面**不需要**每个引用它们的页面都添加反向链接：

| Hub 页面 | 说明 |
|----------|------|
| [[主页]] | 知识库总入口，导航索引 |
| [[术语索引]] | 全量术语速查表 |
| [[规则]] / [[schema/规则\|规则]] | 规则定义 |
| [[欢迎]] | 欢迎页面 |
| [[Agent工作流协议]] | 本页面 |

**Hub 页面的维护义务**：当新增重要页面时，应在 [[主页]] 的导航表中添加条目，在 [[术语索引]] 中注册新术语。

### 3.4 链接验证规则

Agent 在提交任何修改前**必须**执行以下验证：

1. **链接存在性**：所有 `[[wikilinks]]` 必须解析到已存在的页面文件
2. **链接无后缀**：wikilink 中不包含 `.md` 后缀
3. **链接无断链**：不产生 Obsidian 中的红色断链
4. **管道符完整性**：使用管道符的链接，显示文字不为空

若发现链接目标不存在，应：
- 创建对应的 stub 页面（使用 2.2 的 stub 模板），或
- 移除该链接并标注 `<!-- TODO: 待创建 [[目标页面]] -->`

## 四、冲突解决机制

### 4.1 命名冲突

**创建前检查**：Agent 在创建任何新页面前，必须先扫描 `wiki/` 目录，检查是否已存在：
- 完全同名文件
- 语义等价页面（如 `P4P策略.md` 与 `P4P投流策略.md`）

**处理方式**：
- 若已存在语义等价页面 -> **不创建新页面**，在现有页面上增补内容
- 若已存在同名文件但内容不同 -> 使用更精确的命名（如加日期后缀、加领域前缀）
- 若不确定 -> 在 Handoff 信息中标注，交由 Human Editor 裁决

### 4.2 内容冲突

当新数据与已有页面内容冲突时：

1. **新数据优先**：更新的数据覆盖旧数据（如产品数据、运营指标）
2. **保留历史**：在页面末尾或注释中保留旧数据，格式：`<!-- 历史值 (YYYY-MM-DD): 旧值 -->`
3. **标注变更**：在页面末尾的更新日志区域记录变更说明
4. **冲突不确定**：标注 `<!-- CONFLICT: 新旧数据不一致，需人工确认 -->` 并通知 Human Editor

### 4.3 标签冲突

- `schema/规则.md.md` 中的标签分类体系是**唯一权威来源**
- 当页面使用的标签不在标准标签集中时：
  - Agent 应使用最接近的标准标签
  - 在页面中注释建议新增标签：`<!-- TAG_PROPOSAL: 建议新增标签 "xxx" -->`
  - 通知 Human Editor 或 schema-agent 更新规则

### 4.4 术语冲突

- `schema/规则.md.md` 中的术语定义库和 [[术语索引]] 是术语权威
- 发现术语定义不一致时，以 `schema/规则.md.md` 为准
- 如需修正术语定义，须同时更新三个位置：
  1. `schema/规则.md.md` 术语定义表
  2. [[术语索引]] 对应条目
  3. [[主页]] 的快速导航表（如适用）

## 五、Agent 角色定义

### 5.1 LLM Wiki Agent（主力内容生成器）

| 属性 | 说明 |
|------|------|
| **工具** | LLM Wiki 桌面应用 (v0.2.0) |
| **主要职责** | 将 `raw/` 原始资料转化为 `wiki/` 结构化知识页面 |
| **写入权限** | `wiki/` 目录（创建、编辑、删除页面） |
| **只读权限** | `raw/` 目录（读取但不修改）、`schema/` 目录（读取规则） |
| **核心行为** | |
| 新页面创建 | 遵循 2.2 页面模板，包含完整 frontmatter |
| 术语维护 | 识别新术语并更新术语索引和 schema 术语表 |
| 双向链接 | 创建正向链接 + 更新反向链接 |
| 标签生成 | 按标签分类体系为页面打标签 |
| 冲突校验 | 创建前检查命名冲突，编辑时检查内容冲突 |

**约束**：
- 不得修改 `raw/` 目录
- 不得修改 `schema/规则.md.md`（但可建议修改）
- 不得创建与现有页面语义重复的页面
- 不得删除其他 Agent 创建的页面（标记 `stub` + `待完善` 即可）

### 5.2 QoderWork Agent（维护与健康检查）

| 属性 | 说明 |
|------|------|
| **工具** | QoderWork 平台 |
| **主要职责** | 知识库维护、健康检查、链接修复、报告生成 |
| **写入权限** | `wiki/` 目录（编辑现有页面、创建维护报告） |
| **只读权限** | `raw/`、`schema/`、根目录 |
| **核心行为** | |
| 断链检测 | 扫描所有页面的 `[[wikilinks]]`，报告断链 |
| 孤儿页面检测 | 发现无任何链接指向的孤立页面 |
| Frontmatter 校验 | 检查所有页面是否包含必要的 frontmatter 字段 |
| 标签合规检查 | 验证标签是否符合标准分类体系 |
| Stub 页面统计 | 统计 `stub` + `待完善` 标签的页面数量并报告 |
| 健康报告 | 生成知识库健康度评估报告 |

**约束**：
- 不主动创建新的知识页面（仅 LLM Wiki Agent 负责）
- 编辑现有页面时仅修复格式问题、补充缺失链接，不改变核心内容
- 发现严重内容问题应通过 Handoff 协议通知 LLM Wiki Agent 或 Human Editor

## 六、质量检查清单

**触发时机**：任何 Agent 在对 vault 进行修改后，必须执行以下检查。

### 6.1 必检项目（每次修改后）

- [ ] **无断链**：所有 `[[wikilinks]]` 均解析到已存在的页面文件
- [ ] **Frontmatter 完整**：所有新建页面包含 `tags`、`created` 字段
- [ ] **双向链接维护**：新页面的 `## 相关链接` 至少有 2 个链接
- [ ] **反向链接更新**：相关的已有页面已添加回链
- [ ] **标签合规**：所有标签属于标准分类体系
- [ ] **命名无冲突**：无同名或语义等价的重复页面
- [ ] **Wikilink 格式**：无 `.md` 后缀、无反斜杠转义、管道符使用正确

### 6.2 周期性检查（由 QoderWork Agent 执行）

## 经验反馈闭环

### 强制规则

每个 Agent 在执行任何实质性任务时，必须遵循以下三步闭环：

**执行前 — 查经验**:
1. 读取 [[经验索引]]，检查是否有与当前任务相关的历史经验
2. 如有相关经验，读取对应页面，理解前车之鉴
3. 如有相关的"失败复盘"，必须确认当前方案已规避已知陷阱

**执行中 — 记录关键决策**:
1. 记录选择了什么方法、为什么选这个方法
2. 记录考虑过但放弃的替代方案
3. 记录任何意外发现

**执行后 — 沉淀经验**:
1. 按 [[经验记录模板]] 创建经验记录页面（无论成功还是失败）
2. 如果本次经验修正了已有 Wiki 页面的内容，更新对应页面并添加修正说明
3. 将新经验页面链接到 [[经验索引]]

### 豁免条件

以下场景可跳过经验记录：
- 纯读取/查询操作（不修改任何内容）
- 格式修复、链接修复等机械性操作
- 已有多条相同模式的经验记录（无需重复）

### 经验验证等级

| 等级 | 定义 | 使用约束 |
|------|------|----------|
| `verified: true` | 经过 A/B 测试或物理验证 | 可直接作为决策依据 |
| `confidence: high` | 多次复现或深度分析 | 可作为主要参考 |
| `confidence: medium` | 单次经验或推断 | 仅供参考，需自行验证 |
| `confidence: low` | 假设或未经证实的推测 | 仅作参考，不建议直接引用 |
