# 知识库维护 SOP

> 本文档定义了 AIP-LLMWiki 知识库的标准维护操作流程。  
> 维护工作按**日度 / 周度 / 月度**三级节奏执行，覆盖链接健康、内容质量、标签合规、结构优化等维度。  
> 主要执行者为 QoderWork Agent（维护检查）和 LLM Wiki Agent（内容修复），审批者为 Human Editor。  
> 本 SOP 与 [[Agent工作流协议]] 和 [[Agent-Handoff协议]] 配套使用。

---

## 一、日常维护（Daily）

**执行者**: QoderWork Agent（自动）或 Human Editor（手动）  
**触发时机**: 每日首次操作知识库时，或任何 Agent 完成批量修改后  
**预计耗时**: 5-10 分钟

### 1.1 断链扫描

**目标**: 确保所有 `[[wikilinks]]` 均指向存在的页面。

**步骤**:

1. 扫描 `wiki/` 目录下所有 `.md` 文件
2. 提取每个文件中的 `[[wikilinks]]`（含管道符别名的目标文件名部分）
3. 验证每个链接目标是否存在对应的 `.md` 文件
4. 汇总结果

**处理流程**:

```
发现断链
├── 目标页面应当存在但缺失
│   └── 创建 stub 页面（使用 Agent工作流协议 2.2 的 stub 模板）
├── 目标页面名称拼写错误
│   └── 修正 wikilink 为正确的文件名
├── 目标页面已被删除或重命名
│   └── 更新引用方的 wikilink 指向新名称
└── 断链数量 > 10
    └── 生成 P1 Handoff 消息通知 Human Editor
```

**输出**: 当日断链检查报告（格式参照 [[Agent工作流协议]] 6.3）

### 1.2 新页面 Frontmatter 检查

**目标**: 确保当日新建的所有页面包含完整的 frontmatter。

**检查项**:
- [ ] `tags` 字段存在且至少包含 1 个标签
- [ ] `created` 字段存在且日期正确
- [ ] 使用的标签属于标准分类体系（参照 [[Agent工作流协议]] 2.6）

### 1.3 当日变更日志

**目标**: 记录当日所有修改，便于追溯。

**记录位置**: 当日笔记文件（如 `2026-06-08.md`）

**记录格式**:

```markdown
### 知识库变更 — YYYY-MM-DD

| 时间 | Agent | 操作 | 页面 | 摘要 |
|------|-------|------|------|------|
| HH:MM | [Agent] | 新建 | [[页面名]] | [内容概述] |
| HH:MM | [Agent] | 编辑 | [[页面名]] | [修改内容] |
```

## 二、周度复盘（Weekly）

**执行者**: QoderWork Agent 主导，结果提交 Human Editor 审阅  
**触发时机**: 每周日（与 [[Weekly_Review_协议]] 同步）  
**预计耗时**: 15-30 分钟

### 2.1 知识库健康扫描

**目标**: 全面评估知识库的结构健康度。

**扫描项目**:

| 检查项 | 方法 | 通过标准 |
|--------|------|----------|
| 断链总数 | 全量扫描 `[[wikilinks]]` | 0 条断链 |
| 孤儿页面 | 统计无任何入链的页面 | 排除 Hub 页面后，孤儿页 < 5 |
| Stub 页面占比 | 统计 `stub` 标签页面数 | 占比 < 15% |
| Frontmatter 缺失 | 检查无 frontmatter 的页面 | 0 个缺失 |
| 标签不合规 | 比对标签与标准分类体系 | 0 个不合规标签 |
| 主页导航完整性 | 检查重要新页面是否加入 [[主页]] | 100% 覆盖 |
| 术语索引完整性 | 新术语是否已注册到 [[术语索引]] | 100% 注册 |

### 2.2 孤儿页面处理

**定义**: 没有任何其他页面通过 `[[wikilink]]` 链接到的页面（Hub 页面除外）。

**Hub 页面豁免清单**:
- [[主页]]、[[术语索引]]、[[规则]]、[[欢迎]]
- [[Agent工作流协议]]、[[Agent-Handoff协议]]、[[知识库维护SOP]]
- 每日笔记文件（`YYYY-MM-DD.md`）

**处理流程**:

```
发现孤儿页面
├── 页面内容完整且有价值
│   ├── 找到相关的已有页面
│   └── 在相关页面的 "相关链接" 中添加指向孤儿页面的链接
├── 页面内容为 stub
│   └── 标记优先级，列入待完善队列
├── 页面内容重复
│   └── 合并到已有页面，创建重定向
└── 页面已过时
    └── 标记 "过时" 标签，通知 Human Editor 决定是否删除
```

### 2.3 Stub 页面清理

**目标**: 减少 stub/待完善 页面的积压数量。

**步骤**:

1. 统计所有标记 `stub` 或 `待完善` 标签的页面
2. 按创建时间排序，优先处理最早创建的
3. 对于有对应 `raw/` 来源的 stub：
   - 通知 LLM Wiki Agent 处理（通过 [[Agent-Handoff协议]]）
4. 对于无对应来源的 stub：
   - 检查是否仍有必要保留
   - 如不需要，通知 Human Editor 决定是否删除
5. 更新 stub 页面统计数

### 2.4 周度报告

**输出格式**:

```markdown
## 知识库周报 — YYYY-MM-DD (第 N 周)

### 关键指标

| 指标 | 本周 | 上周 | 趋势 |
|------|------|------|------|
| Wiki 页面总数 | N | N | +/- |
| 新建页面 | N | — | — |
| 编辑页面 | N | — | — |
| 断链数量 | N | N | +/- |
| 孤儿页面 | N | N | +/- |
| Stub 页面 | N (X%) | N (X%) | +/- |
| 术语总数 | N | N | +/- |

### 本周工作摘要
- [主要工作内容]

### 发现问题
1. [问题描述] — 处理状态：已修复 / 待处理

### 下周计划
- [待处理事项]
```

## 三、月度审计（Monthly）

**执行者**: QoderWork Agent 生成报告，Human Editor 审批  
**触发时机**: 每月第一个周日  
**预计耗时**: 1-2 小时

### 3.1 全量 Vault 分析

**分析维度**:

| 维度 | 内容 |
|------|------|
| **规模分析** | 页面总数、总字数、平均页面字数 |
| **结构分析** | 各领域页面数量分布（Palantir / LILIS / Alibaba / 独立站 / 通用） |
| **链接分析** | 总链接数、平均入链数、平均出链数、链接密度 |
| **标签分析** | 各标签下页面数量分布、标签使用趋势 |
| **内容深度分析** | 完整页面 vs stub 页面比例、内容质量评分 |
| **活跃度分析** | 过去 30 天内被编辑的页面占比 |

### 3.2 标签清理

**步骤**:

1. 导出全量标签使用统计
2. 识别以下问题标签：
   - **未使用标签**：定义在 schema 中但无页面使用
   - **仅一次标签**：仅被 1 个页面使用的非标准标签
   - **近似标签**：语义重叠的标签对（如 `Agent` 和 `Agent协议`）
   - **非标标签**：不在标准分类体系中的标签
3. 生成标签清理建议：
   - 合并近似标签
   - 删除未使用标签
   - 将仅一次标签升级为标准标签或删除
4. 提交 Human Editor 审批
5. 审批通过后执行清理，更新 `schema/规则.md.md`

### 3.3 过时内容归档

**识别标准**:

| 条件 | 判定 |
|------|------|
| 页面创建超过 90 天且无任何编辑 | 可能过时 |
| 页面引用的 `raw/` 来源已被更新版本替代 | 内容过时 |
| 页面中的数据指标明显偏离当前实际值 | 数据过时 |
| 页面被标记 `verified: false` 超过 60 天 | 待验证 |

**处理流程**:

1. 列出疑似过时页面清单
2. 逐一评估：
   - **仍有效**：更新 `verified: true`，补充最新数据
   - **部分过时**：更新过时部分，保留仍有效的内容
   - **完全过时**：标记 `archived` 标签，移至归档区域或通知 Human Editor 删除
3. 生成归档报告

### 3.4 月度审计报告

**输出格式**:

```markdown
## 知识库月度审计报告 — YYYY年MM月

### 知识库规模

| 指标 | 本月 | 上月 | 变化 |
|------|------|------|------|
| Wiki 页面总数 | N | N | +/-N |
| Raw 文件数 | N | N | +/-N |
| 总字数 | N | N | +/-N |
| 术语总数 | N | N | +/-N |
| 标签总数 | N | N | +/-N |
| 总链接数 | N | N | +/-N |

### 领域分布

| 领域 | 页面数 | 占比 |
|------|--------|------|
| Palantir AIP | N | X% |
| LILIS 业务 | N | X% |

### 标签清理结果

### 过时内容处理
```

## 四、新增知识流程（New Knowledge Ingestion）

### 4.1 知识入库管线

```
原始资料 (外部) 
    │
    ▼
┌─────────────┐
│ 1. Human Editor │ ── 将原始资料放入 raw/ 目录
└─────────────┘
    │
    ▼
┌─────────────┐
│ 2. LLM Wiki     │ ── 读取 raw/ 文件，提取结构化知识
│    Agent        │    创建 wiki/ 页面（遵循页面模板）
└─────────────┘    注册术语到术语索引
    │              添加双向链接
    ▼
┌─────────────┐
│ 3. QoderWork    │ ── 执行质量检查清单
│    Agent        │    验证链接、frontmatter、标签
└─────────────┘    更新反向链接
    │
    ▼
┌─────────────┐
│ 4. Human Editor │ ── 审阅内容准确性
│   (可选)        │    设置 verified: true
└─────────────┘    更新主页导航表
    │
    ▼
  知识入库完成
```

### 4.2 各阶段详细操作

#### 阶段 1：资料接收

**执行者**: Human Editor  
**操作**:
1. 将原始资料文件放入 `raw/` 目录
2. 确保文件命名清晰、格式规范
3. （可选）在当日笔记中记录新增资料

#### 阶段 2：知识提取与页面创建

**执行者**: LLM Wiki Agent  
**操作**:
1. 读取 `raw/` 中的新资料
2. 识别知识点，按领域分类
3. 检查现有 wiki 页面，避免重复
4. 为每个知识点创建 wiki 页面：
   - 遵循 [[Agent工作流协议]] 2.2 页面模板
   - 包含完整 YAML frontmatter
   - 在 `source` 字段引用原始文件
5. 识别新术语，更新 [[术语索引]]
6. 添加正向链接（至少 2 个相关页面）
7. 更新相关已有页面的反向链接
8. 更新 [[主页]] 导航表（如有重要新页面）

#### 阶段 3：质量检查

**执行者**: QoderWork Agent  
**操作**:
1. 执行 [[Agent工作流协议]] 第六章质量检查清单
2. 全量断链扫描
3. Frontmatter 完整性检查
4. 标签合规性验证
5. 生成质量检查报告
6. 如有问题，按 [[Agent-Handoff协议]] 交接给 LLM Wiki Agent 修复

#### 阶段 4：人工审阅（可选）

**执行者**: Human Editor  
**触发条件**: 新建页面 > 5 个，或涉及核心业务逻辑  
**操作**:
1. 审阅新建页面的内容准确性
2. 验证术语定义的正确性
3. 确认页面结构合理
4. 设置 `verified: true`
5. 在审阅完成前，页面保持可见但标记为未验证

### 4.3 快速入库（单文件快速处理）

当只需处理单个 raw/ 文件时，可使用简化流程：

1. LLM Wiki Agent 直接处理，创建 wiki 页面
2. Agent 自行执行质量检查（简化版）
3. 在当日笔记中记录变更
4. 无需生成完整 Handoff 消息

**适用条件**:
- 单次处理 <= 1 个 raw/ 文件
- 新建页面 <= 3 个
- 不涉及核心业务逻辑变更

## 五、应急处理流程

### 5.1 大量断链（断链数 > 20）

```
发现大量断链
├── 判断原因
│   ├── 文件被误删 -> 从 Git 历史恢复（如有版本控制）
│   ├── 批量重命名导致 -> 全量修复 wikilink 引用
│   └── 原因不明 -> 生成 P0 Handoff，通知 Human Editor
├── 紧急修复
│   ├── 优先修复被引用最多的断链目标
│   └── 次要断链创建 stub 页面
└── 事后复盘
    └── 记录在当日笔记中，分析根因
```

### 5.2 页面内容损坏

```
发现页面内容损坏
├── 评估损坏程度
│   ├── 格式损坏（Markdown 语法错误） -> 直接修复
│   ├── 内容丢失 -> 尝试从原始资料重建
│   └── 内容被覆盖 -> 从备份恢复 + 合并新内容
├── 生成 P0 Handoff 通知 Human Editor
└── 记录事件，更新应急处理经验
```

### 5.3 Schema 规则冲突

```
发现 schema 规则冲突
├── 记录冲突详情
├── 以当前 schema/规则.md.md 为准继续工作
├── 生成 P1 Handoff 通知 Human Editor
└── 等待 Human Editor 更新 schema 后同步修改相关页面
```

## 六、维护工具与命令参考

### 6.1 常用扫描命令

**统计 wiki 页面数**:
```bash
ls wiki/ | wc -l
```

**查找 stub 页面**:
```bash
grep -rl "stub" wiki/ --include="*.md"
```

**查找无 frontmatter 的页面**:
```bash
for f in wiki/*.md; do head -1 "$f" | grep -q "^---" || echo "$f"; done
```

**提取所有 wikilinks**:
```bash
grep -oh '\[\[[^]]*\]\]' wiki/*.md | sort | uniq -c | sort -rn
```

### 6.2 健康度评分标准

| 维度 | 10 分 | 8 分 | 6 分 | 4 分 | 2 分 |
|------|-------|------|------|------|------|
| **链接健康度** | 0 断链 | 1-3 断链 | 4-10 断链 | 11-20 断链 | >20 断链 |
| **内容完整度** | stub <5% | stub 5-10% | stub 10-15% | stub 15-25% | stub >25% |
| **标签规范度** | 0 非标 | 1-2 非标 | 3-5 非标 | 6-10 非标 | >10 非标 |
| **结构合理度** | 0 孤儿 | 1-3 孤儿 | 4-8 孤儿 | 9-15 孤儿 | >15 孤儿 |
