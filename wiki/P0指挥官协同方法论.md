# 🏆 P0 级指挥官协同方法论 (P0 Conductor-Executor Protocol)

> **经验基石**：2026-05-23 实测证明，通过 API 直接交付 (P0) 比 UI 模拟 (P2) 效率提升 60 倍以上。未来的任务必须从该正向结果开始，禁止路径退化。

## 1. 核心执行路径：双 Agent 协作 (The Power of Two)

当执行国际站批量任务（上品、优化、调价）时，必须采用以下协同模式：

| 角色 | 实体 | 职能 | 交互方式 |
| :--- | :--- | :--- | :--- |
| **指挥官 (Conductor)** | **Accio** | **注入灵魂**：调取 LLM Wikis (Ontology) 中的品牌、市场、合规约束，生成高精度指令。 | 发起 API 任务请求。 |
| **执行官 (Executor)** | **生意助手 Agent** | **打通手脚**：利用 Native API (原生连接) 权，在阿里后端执行毫秒级批量处理。 | 接收 JSON 指令包并执行。 |

## 2. P0 路径锁定准则 (SPE Strict Rule)

- **正向起点**：后续执行类似“普通品优化”任务，起点必须是 **API 直接调用**。
- **指令封装**：Agent (Accio) 必须将 Wiki 中的文字规范转化为 **结构化 JSON 指令**。例如，将“品牌主张”转化为 `productSellingPoint` 的固定变量。
- **禁止回退**：严禁在已知 API/插件接口存在的情况下启动浏览器模拟 (Browser UI)。

## 3. 物理闭环标准

- **API 状态优先**：以服务器返回的 `type: "success"` 为第一验证点。
- **异步回读确认**：在 API 提交后，必须通过独立会话（如 `get_product_detail`）回读物理状态，确保数据已写入数据库。
